## Fix #1 — ISSUE-051 (Skonto-Inputs-Unmount-Race)

### Mechanism

```
User drueckt Backspace im Prozent-Input bei "2.0":
  Input value=""
    -> SkontoSection.onChange(null, 7)
      -> patchAndSave({skonto_percent: null, skonto_days: 7})
        -> [optimistic] setProposal(prev => {...prev, skonto_percent: null, skonto_days: 7})
          -> Re-Render: isOn = (skonto_percent !== null) = false
            -> {isOn ? <Inputs/> : null} -> Inputs UNMOUNT
              -> Cursor verliert Input -> "geht nach oben"
                -> User kann keine neue Zahl tippen
```

Mein SLC-572-Fix (lastKnownGoodSkontoRef + revert-on-error) ist technisch korrekt — der Toggle bleibt nicht mehr drift-haftig. Aber die `isOn`-Inferenz ist zu reaktiv: jeder transient-null-Optimistic-State macht die Inputs sofort unsichtbar.

### Fix

`isOn = (skonto_percent !== null || skonto_days !== null)` in [skonto-section.tsx](cockpit/src/app/(app)/proposals/%5Bid%5D/edit/skonto-section.tsx).

- **Backspace bei Prozent:** percent=null, days=7 → **isOn=true** → Inputs bleiben gemountet → User tippt "3" → onChange(3, 7) → patchAndSave → debounce-reset → kein invalider Save raus.
- **Toggle-Click OFF:** explizit onChange(null, null) → percent=null AND days=null → **isOn=false** → Inputs unmounten korrekt.
- **Mutex-Auto-Clear:** patchAndSave({null, null}) bei 100%-Vorkasse → beide null → isOn=false → korrektes Verhalten.

DEC-126 / `lastKnownGoodSkontoRef` / `revertPatchIfSkontoFailed` bleiben unangetastet — die Pure-Function-Logik ist OK, nur die Display-Inferenz war zu eng.

### Edge-Case

User Backspace **beide** Felder schnell hintereinander: percent=null, dann days=null → isOn=false → Inputs unmounten. In der Praxis irrelevant, weil:
- Realistisches User-Verhalten: edits ein Feld, geht zum naech.sten.
- debounce-timer resettet bei jedem Keystroke; invalides Save-Pattern feuert nur bei >500ms Pause.
- Falls doch passiert: User klickt Toggle erneut ON → Default-Werte zurueck.

## Fix #2 — ISSUE-052 (PaymentTerms-Dropdown-UUID-Display)

### Mechanism

base-ui's `<Select.Value placeholder="...">` rendert per Default den Raw-Value-String. Wenn der User ein Template waehlt, wird `selectedId = templateId` (UUID), und der Trigger zeigt:

```
[f57bd7b7-c711-4d5f-bbd6-894e63ec38e4 ▼]
```

statt:

```
[Standard 14 Tage netto (Default) ▼]
```

ISSUE-048 (BL-418-Fix) hatte den Initial-Mount-Display gefixt (`useState<string>("")` matched kein SelectItem → Placeholder greift). Aber der Selected-Display-Pfad blieb kaputt.

### Fix

Render-Callback als Function-Child von `<SelectValue>` in [payment-terms-dropdown.tsx](cockpit/src/app/(app)/proposals/%5Bid%5D/edit/payment-terms-dropdown.tsx):

```tsx
function renderSelected(value: string): ReactNode {
  if (!value) return null; // empty -> Placeholder greift
  if (value === CUSTOM_VALUE) return "(eigene Eingabe)";
  const template = templates?.find((t) => t.id === value);
  if (!template) return null;
  return `${template.label}${template.is_default ? " (Default)" : ""}`;
}

<SelectValue placeholder="(eigene Eingabe)">
  {renderSelected}
</SelectValue>
```

base-ui spread propagation: shadcn-Wrapper `<SelectValue>` spreaded `{...props}` zu `<SelectPrimitive.Value>`, daher Function-Child kommt sauber durch.
