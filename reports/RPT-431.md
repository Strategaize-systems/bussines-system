## 4 Open Questions — Aufgeloest

**Q1 — Container-Bootstrap-Pfad?**
→ **DEC-202: Manual-Apply via docker exec, kein Dockerfile-Hook.** Dockerfile-Hook waere 2-3h auf 3-4h-Sprint + Production-Daten-Drift-Risiko. Manual-Apply ist konsistent mit `sql-migration-hetzner.md` und `feedback_ssh_migrations_always_claude.md`. Wenn Staging/CI hinzukommt → BL-475 in V7.3+.

**Q2 — Seed-Daten-Umfang?**
→ **Keine Aenderung.** Existing 50/200/100/500 Records auf 5 Members reicht fuer RLS-Matrix. qa-admin braucht nur Aux-Fixtures (meetings/proposals/email_messages/calls = 4 Records).

**Q3 — Internal-Test-Mode-Gate?**
→ **Entfaellt.** Kein Bootstrap-Hook = kein Auto-Run = kein Gate-Bedarf.

**Q4 — Idempotenz-Pattern?**
→ **Bestehendes DELETE-then-INSERT bleibt.** Practical idempotent. Refactoring auf ON CONFLICT + deterministische UUIDs waere Risiko ggu. bestehenden Tests, die Random-UUID-Verhalten erwarten.

## Security / Privacy Considerations

- **Test-Passwords sind bekannte Strings** — akzeptabel, weil Internal-Test-Mode + TEST_TEAM_ID-Isolation + Production-Admin separat.
- **Production-DB-Modifikation** auf Coolify (Internal-Test-Mode-Pattern) — Sicherheits-Layer:

  - `[TEST]`-prefixed Records (visuell markiert)
  - BEGIN/COMMIT-Transaktion (atomar)
- **Audit-Log** — Seed-Run wird NICHT in `audit_log` geschrieben. Slice-Report-Doku reicht.

## Constraints and Tradeoffs

| Constraint | Tradeoff |
|---|---|
| Kein Bootstrap-Hook | Manual-Apply nach jedem V7.2-Deploy. Akzeptabel weil Re-Seed nur ~1-2x/Quartal noetig. |
| Manual-Apply via docker exec | Agent dokumentiert Run. Konsistent mit Strategaize-Disziplin. |
| Path-Alias via resolve.alias | 4-Zeilen-Patch vs. neue Dep + Plugin-Init-Cost. Bei Multi-Alias V8+ ggf. Plugin-Switch. |
| qa-admin in TEST_TEAM_ID | qa-admin sieht nur TEST-Daten, nicht Strategaize-Production. Akzeptabel weil V7-RLS admin-Policies bereits via richard@bellaerts.de getestet. |
| Time-Box 3-4h | Bei MT-1-Reibung (z.B. POSTGRES_URL-Drift) wird MT-2 unabhaengig vorgezogen. |
