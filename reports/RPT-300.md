# SLC-572 — Skonto-Toggle UI-State-Drift Bugfix (V5.7)

## Outcome Summary

**Status: fully complete (4/4 MTs done).**

Der in V5.6 SLC-562 Live-Smoke (RPT-277) als BL-419 / ISSUE-049 dokumentierte UI-State-Drift-Bug ist gefixt. Nach Save-Error fuer einen Skonto-touching Patch rollt der lokale State auf den letzten vom Server bestaetigten Wert zurueck, sodass der Toggle nicht mehr in einem invaliden Zwischenzustand (z.B. `skonto_percent=null` + `skonto_days=7`) haengen bleibt. Decision-Logic ist als Pure Function extrahiert und mit Vitest abgesichert.

## MT-1 — Investigation + Pattern-Erweiterung-Decision

**Race-Pfad identifiziert:**

1. User aktiviert Skonto-Toggle → DB `{2.0, 7}`, ref `{2.0, 7}`.
2. User loescht Prozent-Input komplett → `SkontoSection.onChange(null, 7)` (Zeilen 90-94 von `skonto-section.tsx`: `raw === "" ? null : Number(raw)`).
3. `patchAndSave({skonto_percent: null, skonto_days: 7})`:
   - Optimistic: `onProposalChange` → workspace-state `skonto_percent = null` → `SkontoSection.isOn = (skonto_percent !== null)` = `false` → Toggle visuell OFF.
   - Debounced: `updateProposal(...)` → Server `validateSkonto(null, 7)` failed (both-or-neither rule) → `{ok: false, error: ...}`.
4. `setSaveStatus('error')` + `setErrorMessage(...)`. **Aber: workspace-state blieb beim invaliden `null`** — kein Revert-Pfad. DB blieb beim alten gueltigen `{2.0, 7}`. Page-Reload heilt's, weil dann Server-Render mit DB-State neu lokal initialisiert wird.

**Pattern-Erweiterung-Decision: out-of-scope.**

- **PaymentTermsDropdown** — nur length-checks (max 2000 Zeichen), realistisch kein Reject-Pattern bei Tipp-Drift.
- **SplitPlanSection** — `persistMilestones` (in `proposal-workspace.tsx` Zeilen 132-139) macht Pre-flight-Validation: bei invaliden Plan KEIN Server-Call. Damit existiert dort der Race nicht.
- Conclusion: nur SkontoSection braucht den Fix. PaymentTerms + SplitPlan in QA nur regression-frei verifizieren.

## MT-2 — useRef + Revert-Logic in proposal-editor.tsx

`lastKnownGoodSkontoRef = useRef<SkontoState>({skonto_percent, skonto_days})` initialisiert mit den aktuellen `proposal.*`-Werten beim Erst-Render.

`persistPatch(patch)` erweitert um:
- **Auf Success:** `lastKnownGoodSkontoRef.current = nextSkontoRefAfterSave(prev, patch)` — aktualisiert nur, wenn der Patch Skonto beruehrt hat.
- **Auf Error:** `revert = revertPatchIfSkontoFailed(patch, lastKnownGoodSkontoRef.current)` — bei skonto-touching Patch ein Revert-Patch zurueck, sonst `null`. Wenn `revert !== null`: `onProposalChange(revert)` → workspace-state-Spread merged Skonto-Felder zurueck auf last-good. Andere Felder bleiben unangetastet (AC6).

`useCallback`-Deps: `[proposal.id, onProposalChange]` — beide stabil aus dem Workspace.

Anti-Pattern bewusst NICHT implementiert: useEffect-Sync `proposal.skonto_*` → ref. Das wuerde im aktuellen Workspace-Modell falsch funktionieren — `proposal` State updatet ueber `onProposalChange` (optimistic), nicht ueber Server-Response, daher wuerde useEffect die ref auf den invaliden optimistischen Wert syncen, was den Revert ad absurdum fuehrt. ref bleibt deshalb nur via Save-Success aktuell.
