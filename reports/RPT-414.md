# RPT-414 — V7.1 Requirements

## 1. Summary

V7.1 ist als Polish-Sprint definiert mit 3 Features (FEAT-711 + FEAT-712 + FEAT-713) aus den drei offenen BLs nach V7-Release (BL-469 + BL-468 + BL-466). Sprint-Scope nach User-Entscheidung 2026-05-15 (3 Scope-Fragen geklaert):

1. **Permission-Cut:** Admin alles / Admin+Teamlead Workflow+Templates+Kampagnen+Team / Member nur Profil+Working-Hours+Meeting.
2. **Drilldown-Scope:** Pipeline + Aktivitaeten + Mein-Tag (alle 3 Sub-Pages voll erweitert).
3. **Reihenfolge:** BL-469 → BL-468 → BL-466 (sichtbar zuerst).

## 2. Goal

Sichtbare User-Wuensche zuerst (Settings-Permissions + Drilldown-Vollausbau), dann V7-Defense-in-Depth-Polish. V7 wird damit production-readier fuer ersten Drittnutzer-Test. Internal-Test-Mode bleibt formell aktiv bis Pre-Production-Compliance-Gate.

## 3. Primary users

- **Admin** — verwaltet organisationsweite Settings ohne Risiko durch versehentliche Teamlead/Member-Zugriffe
- **Teamlead** — operative Settings (Workflow/Templates/Kampagnen) + voller Drilldown-Coaching-View
- **Member** — nur eigenes Profil + Working-Hours + Meeting-Settings sichtbar

## 4. V1 Scope (3 Features)

| ID | Feature | Type | Prio | BL | Aufwand |
|---|---|---|---|---|---|
| FEAT-711 | Settings-Permission-Layer (rollen-basiert Admin/Teamlead/Member) | new | high | BL-469 | ~4-6h |
| FEAT-712 | Drilldown-View Vollausbau (Pipeline + Aktivitaeten + Mein-Tag) | improvement | medium | BL-468 | ~5-8h |
| FEAT-713 | V7-Defense-in-Depth Polish (4 Guards + Audit-Doc-Sync) | improvement | medium | BL-466 | ~30 min - 1h |

**Permission-Matrix (FEAT-711) ist im PRD V7.1-Section vollstaendig ausgeschrieben.**

## 5. Out of Scope

- **ISSUE-066 AsyncLocalStorage-Drilldown-Gap-Fix** bleibt fuer V7.5 (Middleware-Pfad-Check)
- **Audit-Trail-Erweiterung auf alle Settings-Aenderungen** — V7.2-Item falls Bedarf
- **Granulare Settings-Audit-Reports** — V7.6 Custom-Reports
- **Custom-Permission-Layer** (User-defined Roles) — bleibt out-of-scope bis konkreter Drittnutzer-Bedarf
- **Bulk-Settings-Import/Export** — kein Bedarf

## 6. Constraints

- Internal-Test-Mode bleibt aktiv (Compliance-Gate ist V7.5+ Thema)
- Keine neuen Provider, keine neue Infrastruktur, keine neue Datenbank
- Keine Schema-Migration zwingend — V7.1 nutzt existierende `profiles.role`-Spalte + Read-Only-Context aus SLC-706 + V7-Helper-Functions aus MIG-034
- TDD-Mandate fuer FEAT-711-Permission-Guards (Live-DB-Tests gegen Coolify-DB)
- Pattern-Reuse-First (CLAUDE.md Core-Default #5): existierende Pages reusen, NICHT Drilldown-Variante neu schreiben

## 7. Risks / Assumptions

- **Risk:** FEAT-712 Drilldown-Vollausbau triggert Filter-State-Cross-Pollution wenn Page-Component naiv wiederverwendet — Mitigation: Filter-Persist-Key mit `viewAsUserId`-Prefix
- **Risk:** FEAT-711 versteckt Sidebar-Eintraege, aber Direct-URL-Navigation muss als 403/Redirect blocken
- **Risk:** FEAT-713 ISSUE-070-Guard auf `bulk-reassign-actions.ts` greift heute nicht via Page-Render (ISSUE-066 AsyncLocalStorage-Gap) — Vitest-Mock-Layer notwendig
- **Assumption:** `assertRole`-Helper existiert bereits oder ist trivial via SLC-702-Pattern erstellbar
- **Assumption:** Pipeline-/Aktivitaeten-/Mein-Tag-Components haben bereits owner_user_id-Filter (sonst FEAT-712 muss Filter-UI ergaenzen)

## 8. Success Criteria

- **FEAT-711 PASS:** Member-Login auf `/settings/branding` → 403/Redirect. Sidebar zeigt rollen-spezifische Sub-Pages. Live-Smoke mit 3 Rollen.
- **FEAT-712 PASS:** `/team/[user]/pipeline` zeigt volle Toggle-Funktionalitaet wie `/pipeline`, alle Mutate-Buttons hidden. Analog `/aktivitaeten` + `/mein-tag`. Filter-State persistiert nicht cross-User.
- **FEAT-713 PASS:** 4 Vitest-Mock-Tests gruen, AUDIT_SERVER_ACTIONS_V7.md syncronisiert, `npm run test:all` 100% gruen.
- **Gesamt-V7.1 PASS:** alle 3 Slices live verifiziert, `npm run test:all` gruen, audit_log-Trail aktiv.
