# V8.15 SLC-913 MT-6 — Login-Rate-Limit-Haertung

## Outcome
ISSUE-120 (Medium) geschlossen (App-Layer-only, keine Migration → resolved ohne /deploy-Gate). DEC-303.

## R-913-3 BLOCKING live-verifiziert (vor jeder Code-Aenderung)
SSH 91.98.20.191 → Throwaway-`traefik/whoami` hinter coolify-proxy, Request mit `X-Forwarded-For: 9.9.9.9` + `X-Real-Ip: 8.8.8.8`. Backend empfing `X-Forwarded-For: <reale Peer-IP>` — der gespoofte `9.9.9.9` wurde **gedroppt, nicht angehaengt**. Traefik laeuft ohne `forwardedHeaders.trustedIPs` (Launch-Args inspiziert) → es VERWIRFT den client-XFF und ersetzt ihn durch die reale Peer-IP (Einzelwert). Die ISSUE-120-Praemisse ("XFF-Rotation → frischer Bucket") greift in dieser Config NICHT. Echte Restluecke: der `login:<email>:<ip>`-Key ist pro-IP — ein Botnet mit vielen REALEN IPs zieht pro IP einen frischen Bucket.

## Fix
- `extractClientIp` nimmt den rechtesten(-Offset) XFF-Eintrag (env `TRUSTED_PROXY_COUNT`, Default 1) statt des linkesten — robust/portabel falls je eine 2. Proxy-Schicht ergaenzt wird; identisch korrekt im heutigen Single-Value-Fall.
- **Primaerer Fix:** IP-unabhaengiger account-scoped Lockout `login-acct:<email>` (20/15min) zusaetzlich zum per-(email,ip)-Limit (5/15min). Peek beider Buckets VOR signInWithPassword (kein GoTrue-Touch wenn gesperrt), Increment beider bei Fail, Clear beider bei Erfolg.
- In-Memory-Counter bleibt (R-912-4 Single-Container); persistenter Store als Folge-Item dokumentiert.
