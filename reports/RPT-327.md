### MT-4 RC-Drift-Fix Race-Path (AC4) — Server-Reject Defense-in-Depth
- **Wichtiger Befund**: V5.7 `useReverseChargeEligibility`-Hook blockt Toggle proaktiv (`disabled=true`) bei fehlender Branding-vat_id. Initialer Test-Setup "Toggle ON ohne vat_id direkt klicken" ist daher **client-side gar nicht reproduzierbar**.
- **Race-Path-Test umgesetzt** (genau das Szenario, das MT-4-Code abdecken soll):
  1. Editor mit gueltiger Branding-vat_id geladen → Toggle enabled
  2. RC ON → OFF Click (regulaer, akzeptiert) → DB: `reverse_charge=false, tax_rate=21`
  3. SQL `UPDATE branding_settings SET vat_id = NULL` mid-session **ohne** Browser-Reload → Editor-Eligibility-Hook nicht reaktiv → Toggle bleibt enabled (`disabled=false`)
  4. Toggle ON Click → Server `validateReverseCharge` Pfad 2 (`!branding.vat_id`) muss rejecten
  5. **DB-Verify**: `reverse_charge=false, tax_rate=21.00` — ON-Versuch wurde **nicht** persistiert. Audit-Log zeigt keinen ON-Eintrag nach 14:59:13. Server hat sauber rejected.
- **UI Toast + Visual-Revert** konnten **nicht direkt** abgetastet werden (Browser-Crash waehrend kontinuierlichem JS-Sampling-Promise nach dem Click). DB-Beweis + Code-Coverage (20 Vitest in `reverse-charge-revert.test.ts`) belegen den Defense-in-Depth-Fix.
- **Restore**: `branding.vat_id` zurueck auf `NL859123456B01`, `proposals.reverse_charge=true, tax_rate=0` via direktem postgres-UPDATE wiederhergestellt (kein Audit-Eintrag fuer Test-Restore — bewusst, vermeidet Schein-Eintraege im History-Log).
