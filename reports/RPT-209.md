## Problems found

1. **Test-Framework fehlte komplett** — Slice-Planning hatte das nicht antizipiert. Per Deviation Rule 4 dem User vorgelegt, Option A gewaehlt. Vitest@4.1.5 als devDep installiert, vitest.config.ts angelegt, package.json scripts ergaenzt. **Skill-Improvement-Kandidat:** `/slice-planning` und `/architecture` sollten Test-Framework-Existenz pruefen, wenn ein Slice Tests fordert (siehe Recommended next step → IMP).

2. **NPM Audit Vulnerabilities** — `npm install -D vitest` meldet 5 Vulnerabilities (4 moderate, 1 high). Ohne Detail-Analyse: typischerweise transitive Dev-Deps. Out-of-Scope fuer SLC-522 — kann in eigenem Tech-Debt-Slice geprueft werden.

3. **TypeScript-Type-Issue im Test** — `let receivedConfig: { apiVersion: string } | null = null` mit Re-Assignment im Closure produzierte `never`-Inferenz. Pattern auf `const captured: { apiVersion?: string } = {}` umgestellt. Fix inline gemacht, nicht als Issue dokumentiert.

4. **AZURE_OPENAI_API_VERSION Whisper-Kompatibilitaet** — Slice-Spec markiert als Risk: API-Version `2024-06-01` koennte fuer Whisper-Endpoints anders heissen. Mein Adapter nutzt diesen Default. Verifizierung gegen echte Microsoft-Doku oder Live-Endpoint ist erst im Pre-Go-Live moeglich. Fallback: ENV-Override existiert. Nicht-Blocker fuer V5.2.

## Open points / deferred decisions

- **Live-E2E-Verifikation gegen Azure** — nicht moeglich in V5.2 (kein Account). Erst beim Pre-Go-Live-Switch sichtbar, ob Azure-API-Drift existiert. Mitigation: Tests sind so genau wie moeglich gegen Microsoft-Doku gebaut.
