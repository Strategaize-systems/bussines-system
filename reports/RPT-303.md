# V5.7 Follow-up #2 — Skonto-Edit-Flicker

## Outcome Summary

Im 2. Live-Smoke (User 2026-05-05 gegen `64c0178`) wurde nach den beiden RPT-302-Fixes (ISSUE-051 + ISSUE-052) ein dritter Bug entdeckt:

> "wenn ich backspace bei 2% mache, erscheint sehr kurz hinweiss 'Beide Skonto-Felden muessen gesetzt sein' und dann wieder 2. Das muss anders. Ich soll nicht speichern koennen mit leeres Feld aber genug Zeit bekommen was auszufuellen."

Inputs blieben (dank ISSUE-051-Fix) gemountet, aber der **debounced Server-Save** feuerte nach 500ms mit dem invaliden Zwischenzustand `{percent: null, days: 7}`. Server rejected, mein SLC-572-Revert rollte auf last-known-good zurueck. Resultat: Error-Flicker plus User kann nicht weiter tippen.

**Fix:** Client-side Validation-Gate vor dem Server-Save. Wenn der debounced Patch einen Skonto-touching Patch traegt und `validateSkonto` rejected, **kein Server-Save** ausgefuehrt. SkontoSection zeigt die Inline-Validation-Message; SaveIndicator bleibt am letzten erfolgreichen Stand. Sobald der User einen validen Wert tippt (z.B. nach Backspace die "3"), feuert der naechste Save mit gueltigem Patch durch.

## Mechanism

```
Vorher (commit 64c0178):
  User Backspace 2% in Prozent-Input
    -> onChange(null, 7) sofort
      -> patchAndSave({percent: null, days: 7})
        -> [optimistic] state {null, 7}
          -> SkontoSection: isOn=true (ISSUE-051 Fix), Inputs gemountet
            -> Inline-Validation-Message: "Beide Skonto-Felder muessen gesetzt sein"
        -> debouncedPersist queued mit 500ms
  ... [User tippt nicht weiter, oder zu langsam] ...
  500ms+:
    -> persistPatch({null, 7})
      -> updateProposal({null, 7})
        -> Server validateSkonto(null, 7) rejected
          -> {ok: false, error: "Beide Skonto-Felder muessen gesetzt sein"}
            -> setSaveStatus("error") + revert auf last-good {2, 7}
              -> Inline-Validation-Message verschwindet (state {2, 7} ist valid)
                -> User sieht: kurzer Error-Flicker, dann "2" zurueck.
                  -> Edit-Versuch unterbrochen, User kann nicht tippen.

Nach (dieser Patch):
  ... [gleicher Pfad bis debouncedPersist] ...
  500ms+:
    -> persistPatch({null, 7})
      -> Client-side validateSkonto(null, 7) rejected
        -> return early (kein Server-Call)
          -> state bleibt optimistisch {null, 7}
            -> SkontoSection zeigt Inline-Validation-Message konstant
              -> SaveIndicator bleibt "Gespeichert" (last good)
                -> User kann frei tippen.
  User tippt "3":
    -> onChange(3, 7)
      -> patchAndSave({percent: 3, days: 7})
        -> debouncedPersist queued (timer reset)
  500ms+:
    -> persistPatch({3, 7})
      -> Client-side validateSkonto(3, 7) ok
        -> updateProposal({3, 7}) -> Server ok
          -> setSaveStatus("saved") + ref aktualisiert auf {3, 7}
```

## Why this is the right fix (and not just longer debounce)

**Alternative A — debounce-Zeit verlaengern (z.B. 500ms → 2000ms):**
- Macht das Symptom seltener, aber nicht weg.
- Skaliert nicht mit User-Tipp-Geschwindigkeit (Senioren mit Two-Finger-Typing sehen weiter Flicker).
- Verschlechtert das Auto-Save-UX fuer alle anderen Felder (ein Debounce gilt fuer alle Patches).

**Alternative B — Validation-Gate (gewaehlt):**
- Server-Save feuert NUR fuer valide States.
- Inline-Validation-Message zeigt User-feedback fuer invalide States.
- Auto-Save-UX fuer andere Felder bleibt 500ms (gefuehlt schnell).
- DEC-126 / Revert-on-Server-Error bleibt als Defense-in-Depth.

**Alternative C — Validation server-side soft (Constraint relaxen):**
- Bricht den `proposals_skonto_both_or_none` DB-CHECK-Constraint.
- Inkonsistenz zwischen Client- und DB-Logik. Schlechte Idee.
