## Findings

### Blocker

**ISSUE-001 — Dockerfile für Cockpit fehlt**
- `/cockpit/Dockerfile` existiert nicht, wird aber in `docker-compose.yml` Zeile 25 referenziert (`build: context: ./cockpit, dockerfile: Dockerfile`)
- Docker Compose Stack kann ohne Dockerfile nicht gebaut werden
- Betrifft: SLC-002, SLC-011

**ISSUE-002 — Dockerfile.kong für Kong Env-Var-Substitution fehlt**
- Kong 2.x ersetzt `${ENV_VAR}` in `kong.yml` NICHT automatisch. Das Blueprint-Projekt löst das über ein `config/Dockerfile.kong` das `envsubst` ausführt.
- `docker-compose.yml` referenziert aber `image: kong:2.8.1` direkt statt einem Build-Context
- Ohne Fix: Kong startet, aber Auth-Keys werden nicht eingesetzt → API komplett offen oder komplett blockiert
- Betrifft: SLC-002, SLC-011

### High

**ISSUE-003 — Supabase DB Image nutzt keinen Build-Context für SQL-Init**
- `docker-compose.yml` mapped `./sql:/docker-entrypoint-initdb.d` als Volume
- Das Supabase-Postgres-Image (`supabase/postgres:15.6.1.145`) hat aber bereits eigene Init-Scripts in `/docker-entrypoint-initdb.d/`
- Das Volume-Mount überschreibt diese → Supabase-interne Rollen (supabase_admin, supabase_auth_admin, authenticator, anon etc.) werden NICHT angelegt
- Das Blueprint-Projekt löst das mit `sql/Dockerfile.db` das zuerst die Supabase-Basis-Rollen anlegt und dann die App-Schemas
- Ohne Fix: PostgREST, GoTrue und Storage können sich nicht authentifizieren
- Betrifft: SLC-002

**ISSUE-004 — Kontakt-Bearbeiten (updateContact) ist nicht in der UI verdrahtet**
- `updateContact` Server Action existiert in `contacts/actions.ts`
- Aber keine UI nutzt sie — Kontakt-Detail hat keinen "Bearbeiten"-Button
- `ContactForm` + `ContactSheet` sind nur auf `createContact` verdrahtet
- Akzeptanzkriterium "Kontakte bearbeiten" (FEAT-001) ist nicht erfüllt
- Betrifft: SLC-003

**ISSUE-005 — Firmen-Bearbeiten (updateCompany) nicht in UI verdrahtet**
- Gleich wie ISSUE-004 für Firmen
- `CompanyForm` existiert, aber kein Edit-Sheet oder Edit-Seite
- Betrifft: SLC-003

**ISSUE-006 — Skills nicht als Claude Code Slash-Commands registriert**
- Die 8 SKILL.md Dateien liegen unter `/skills/marketing/`, `/skills/leads/`, `/skills/pipeline/`
- Aber Claude Code erwartet Skills unter `.claude/skills/` Verzeichnis mit einer bestimmten Struktur
- Ohne korrekte Registrierung sind die Skills nicht als `/skill-name` aufrufbar
- Akzeptanzkriterium "Skills sind über Claude Code als `/skill-name` aufrufbar" (FEAT-004) ist unsicher
- Betrifft: SLC-008, SLC-009

### Medium

**ISSUE-007 — Kontakt-Detail Stub für Aktivitäten**
- `contacts/[id]/page.tsx` Zeile 128: `{/* Placeholder for Activities (SLC-005) and Documents (SLC-005) */}`
- Expliziter Stub — bewusst so geplant, aber dennoch ein Platzhalter im Production-Code
- Akzeptabel da SLC-005 geplant ist, aber sollte nach SLC-005 entfernt werden

**ISSUE-008 — Kontakt-Formular nutzt native `<select>` statt shadcn Select**
- `contact-form.tsx` nutzt ein natives `<select>` für Firma-Auswahl
- Visuell nicht konsistent mit dem Rest der shadcn/ui Oberfläche
- Funktional korrekt, aber UI-Konsistenz-Verstoß

**ISSUE-009 — Keine Fehlermeldungen bei CRUD-Operationen**
- Server Actions geben `{ error: string }` zurück, aber nur Login-Formular zeigt den Fehler an
- Contact-Sheet und Company-Sheet prüfen `result.error` für close-Logik, zeigen aber keinen Fehler-Text
- Bei fehlgeschlagenem Create/Update sieht der User kein Feedback

### Low

**ISSUE-010 — Dashboard ist nur Stub**
- `/dashboard/page.tsx` zeigt nur "Willkommen im Business Cockpit"
- Bewusst geplant (SLC-006), aber die aktuelle Seite nach Login ist leer
- Kein funktionaler Impact

**ISSUE-011 — .env.local enthält Platzhalter-Werte**
- Werte wie `placeholder-will-be-replaced` — akzeptabel für Dev, aber darf nie deployed werden
- Akzeptiert da `.env.local` in `.gitignore` steht

## Offene Risiken

1. **Keine Runtime-Verifikation**: Kein einziger Slice wurde gegen eine laufende Supabase-Instanz getestet. Build-Erfolg ≠ Runtime-Erfolg.
2. **Docker Compose hat 3 Blocker**: Ohne Dockerfile (Cockpit), Dockerfile.kong (Env-Substitution), Dockerfile.db (Supabase Rollen) kann der Stack nicht starten.
3. **Edit-Funktionalität fehlt**: CRUD ist eigentlich CR_D — das U (Update) hat keine UI.
