## Problems Found

1. **Tooling-Pre-Step war noetig (devDeps + vitest-config-Umbau).**
   Die Slice-Spec geht von einem Test-Setup mit React-Component-Tests (RTL + jsdom) aus, das im Repo nicht existierte. Vitest war auf `environment: "node"` und `include: ["src/**/*.test.ts"]` (kein .tsx). Vorgehen nach Rueckfrage an User (Option B "Slice-Spec strict"): npm install --save-dev `@testing-library/react`, `@testing-library/dom`, `@testing-library/jest-dom`, `jsdom`, `@vitejs/plugin-react`; vitest.config.ts auf `jsdom` env + include `*.test.tsx` + plugin-react; vitest.setup.ts mit `import "@testing-library/jest-dom/vitest"` + RTL `cleanup` in `afterEach`. Volltest danach 33/33 -> 38/38 (5 neue ki-workspace-Files), 444/444 -> 466/466 (22 neue Tests). Keine Regression auf vorher gruene Tests.

2. **Markdown-Library war nicht im Repo (Slice-Spec-Annahme falsch).**
   Slice-Spec MT-6 Reuse-Note: "vermutlich `react-markdown` oder Custom — pruefen waehrend Implementation. Fallback: dangerouslySetInnerHTML mit DOMPurify (vorhanden im V5.5 PDF-Renderer)". Pruefung ergab: weder `react-markdown` noch `markdown-to-jsx` noch `DOMPurify` in package.json. pdfmake erwartet eigenes JSON-Format, nicht Markdown. Loesung: kleiner Inline-Markdown-Renderer als Teil von AnswerPane.tsx (parseBlocks + renderInline). Subset: h1/h2/h3, ul-Listen, **bold**, *italic*, `code`, [link](url) mit Schema-Allowlist (`https?:|mailto:|/`). HTML wird durch React-Text-Children automatisch escapet. Keine neue Dep noetig.

3. **Slice-Spec-Deviation: dynamic-import-by-path ersetzt durch loadRunner-Injection-Point.**
   Slice-Spec MT-5: "dynamic import von `report.serverActionPath`, call `runReport(scope)`". Webpack/Turbopack-Probleme bei String-getriebenen dynamic imports (kein Statisches Bundling), zudem schwer testbar mit vi.mock. Statt dessen: `useReportRun` akzeptiert optionalen `loadRunner: (path) => Promise<ReportRunner>`. Default loadt `_mock` per statischem Import. SLC-662/664/666-Caller registrieren echte Runner ueber denselben Hook. Funktional aequivalent, Webpack-friendlich, testbar ohne vi.mock-Tricks. Dokumentiert im MT-5-Commit (`9f38e30`).

4. **Slice-Spec-Deviation: Hook-API errors-via-state statt errors-via-throw.**
   Erste useVoiceCapture-Implementation hat bei Fehlern geworfen + State gesetzt. Tests waren flaky (act-async-Race verschluckt setError). Refactor: Errors landen NUR im error-State, Methoden returnen ohne throw. Cleaner Caller-Pattern, Tests stabil. Ist Hook-API-Aenderung gegenueber Slice-Spec, aber gleicher Funktionsumfang.

5. **Pre-existing TSC-Errors im Repo (nicht von SLC-661).**
   `npx tsc --noEmit` meldet Errors in `src/lib/proposal/{reverse-charge-revert,skonto-revert}.test.ts` und `src/lib/validation/vies-client.test.ts`. Beide pre-existing (V5.7-Carryover), kein Touch in SLC-661. `npm run build` (Next.js + SWC, separater Pfad) ist clean.

6. **Pre-existing Lint-Volume im Repo (168 Errors, 65 Warnings).**
   Auf SLC-661-Files: nur Warnings fuer underscore-prefixed unused parameters in Test-Mocks (`_stream`, `_args`, `_timeslice`). Slice-Spec AC10 verlangt "keine neuen Errors" — erfuellt.

7. **MOCK_PATH konstant `@/lib/ki-workspace/reports/_mock` fuer alle 16 Berichte.**
   In SLC-661 zeigen alle 16 Reports in der Registry auf das gleiche Mock-Modul (per Slice-Spec MT-1). SLC-662/664/666 swappen die echten Pfade (z.B. `@/lib/ki-workspace/reports/tagesanalyse`) ein. Idee: Cache-Key disambiguiert ueber `reportId`, nicht ueber `serverActionPath`.
