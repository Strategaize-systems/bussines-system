## Problems found

1. **Auto-Save bei valid→invalid→valid-Sequenz mit Stale-State-Risiko (klein):** Wenn User 50/50 (valid) → 50/40 (invalid) → 60/40 (valid) tippt und alle 3 Aenderungen innerhalb von 500ms passieren, fired die debounced Save mit dem letzten valid-State (60/40). Korrekt. Wenn User aber 50/50 (valid) → 50/40 (invalid) tippt und stehen bleibt, schedule die Debounce mit invalid; die Validation in `persistMilestones` skippt. Server bleibt auf letzten valid-Stand (50/50). Akzeptiert — UI zeigt SumIndicator-Diff.
2. **Toast-Loesung als Banner-Hinweis (Slice sagt "Toast"):** Slice-Spec MT-6 fordert "Toast-Hinweis 'Skonto deaktiviert (Vorkasse erkannt)'". Kein `sonner`/Toast-Lib im Repo (keine `toast.`-Calls gefunden). Stattdessen: persistente amber-Banner-Sektion ueber der SkontoSection (aktiv solange skontoMutex true). Information ist gleich, UX-Form anders. Per Deviation-Rule 1 (Bug-Fix-aequivalent: kein Toast-Lib vorhanden = nicht-blockierende Loesung). Falls User explizit Toast moechte → V5.7+ Toast-Lib integration als Followup.
3. **Browser/Live-Smoke offen (MT-9):** Kein lokaler Browser-Smoke, kein Hetzner-Live-Smoke. Wird in /qa SLC-563 nach User-Coolify-Redeploy gemacht (Konvention: Frontend liefert Code-complete, /qa verifiziert vor Records-`done`).
4. **`amount`-Snapshot wird beim Workspace-Save berechnet, nicht beim Editor-Render:** `totalGross` wird im Workspace via `useMemo(calculateTotals(items, taxRate))` berechnet und in jedem `handleMilestonesChange`-Call uebergeben. Wenn User Position aendert (`items` wird via debouncedSave nach 500ms persistiert) und parallel Milestone-percent aendert, basiert die naechste Milestone-Save auf dem neuen Total. Akzeptiert per V5.6-Design (siehe Slice-Risk "amount-Stale" — out-of-scope).
