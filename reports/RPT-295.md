- **Browser-Smoke MT-7 offen.** User muss Coolify redeployen. Anschliessend /qa MT-7 mit DevTools-Network-Patch (z.B. `{reverse_charge: true, tax_rate: 21}` direkt an Server-Action posten) und `/audit-log` zur Verifikation der Reject-Fehlermeldung + Audit-Eintraege.
- **DB-Integration-Test gegen Coolify-DB nicht aufgesetzt.** Slice-Spec MT-7-Verification erwaehnt "Vitest gegen Coolify-DB mit RLS-Pattern". Bewusste Reduktion: MT-7-Logic ist Server-Side-Code-only (keine RLS-Policy-Aenderung); DB-CHECK fuer RC↔tax_rate=0 ist via MIG-028 + RPT-291 bereits live verifiziert. Cross-Field-Validation (vat_id, country) ist server-side only und wird durch 14 Pure-Function-Tests abgedeckt. Live-Browser-Smoke ist die Integration-Verifikation. Akzeptables Restrisiko fuer Internal-Test-Mode.
- **Interaktion mit `getProposalForEdit`:** Editor laedt branding+company beim Mount. Server-Action laedt sie erneut bei Save. Bei sehr schnellen Branding-Aenderungen waehrend Editor offen ist konnte der Editor ein altes branding-State zeigen, der Server aber bereits den neuen Wert validieren — Server gewinnt, was die korrekte Semantik ist. Edge-Case akzeptabel.
- **Fire-and-forget-Audit:** `logAudit` ist via `void`-Operator entkoppelt. Bei Audit-Fehler bleibt das Save erfolgreich (Audit-Luecke akzeptabel, doppelter Save nicht).
- **Audit-Eintrag bei jedem Save mit RC=true unveraendert:** Wir feuern den `reverse_charge_toggled`-Audit nur, wenn der Wert tatsaechlich kippt. Wenn der User mehrfach `Save` ausloest mit RC=true unveraendert, wird kein Spam erzeugt. Verify-Path: `parsed.data.reverse_charge !== before.reverse_charge`.
