## Findings

### Medium

#### M-1: Server-Action-RLS-Bypass via createAdminClient (pre-existing, nicht SLC-901-Regression)

**Severity:** Medium (Defense-in-Depth-Verlust, funktional sicher via manuelle `.eq("user_id", user.id)`-Filter)

**Location:**
- `cockpit/src/app/actions/goals.ts` (8x `createAdminClient()` Verwendung, L46/85/130/175/...)
- `cockpit/src/app/actions/kpi-snapshots.ts` (2x `createAdminClient()`, L33/76)
- `cockpit/src/app/actions/activity-kpis.ts` (6x `createAdminClient()`, L20/45/86/106/150/199)

**Beschreibung:** 3 von 4 Klasse-A-Server-Actions fuehren `supabase.auth.getUser()` zur Auth-Pruefung aus, aber alle DB-Operationen via `createAdminClient()` (service_role mit BYPASSRLS). Die neue Klasse-A-Policy `user_id = auth.uid() OR is_admin()` greift fuer diese Pfade NICHT — RLS wird via service_role umgangen. Schutz besteht ausschliesslich durch manuelle `.eq("user_id", user.id)`-Filter im Application-Code.

**Belege:**
```typescript
// goals.ts L40-50
const supabase = await createClient();
const { data: { user } } = await supabase.auth.getUser();
if (!user) return [];
const admin = createAdminClient();   // ← Bypass!
let query = admin.from("goals").select("*").eq("user_id", user.id);
```

```typescript
// goals.ts L85-99
const admin = createAdminClient();   // ← Bypass!
const { data, error } = await admin.from("goals").insert({
  user_id: user.id,
  // ...
});
```

**Discrepancy zur Dokumentation:** `docs/AUDIT_CRON_V811.md` Server-Actions-Tabelle listet diese 3 Files als `Auth-Pfad: authenticated | RLS-Compliance: RLS Klasse-A aktiv`. **Faktisch sind alle 3 RLS-Bypass-Pfade.** Nur `cockpit/src/app/actions/user-settings.ts` (4. Klasse-A-Server-Action) ist Spec-konform — nutzt durchgehend authenticated `createClient()`.

**Begruendung Severity Medium (nicht High):**
- KEIN SLC-901-Regression — Pattern existiert pre-V8.11 und wurde nicht durch SLC-901-Code-Change neu eingefuehrt
- Funktional sicher solange `.eq("user_id", user.id)`-Filter konsequent gesetzt — manuelles Code-Review zeigt: alle 16 admin-Calls in goals.ts/kpi-snapshots.ts/activity-kpis.ts haben den Filter
- Live-Smoke Pfad 3b+3c (Cross-User-Read/Write blockiert) testet die DB-RLS-Engine direkt — der service_role-Bypass-Pfad wuerde Cross-User-Leaks erlauben wenn der manuelle Filter vergessen wuerde
- V8.11-Scope ist Policy-Migration, NICHT Server-Action-Refactor

**Empfehlung:** Cross-Klasse-Beobachtung — pruefen ob B/C/D/E-Klassen-Tabellen das gleiche Pattern haben. Wenn ja: Defense-in-Depth-Refactor als eigene V9 oder V8.12 Slice spaeter. Dev-System-IMP siehe Open Points.
