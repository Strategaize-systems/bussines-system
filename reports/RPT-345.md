# Architecture Report — V6.5 Hintergrund-Sprint

## Architecture Summary

V6.5 ist ein leichter Hintergrund-Sprint OHNE neue Service-Container, OHNE neue externe Integrationen ueber VIES hinaus, OHNE Auth/Session-Aenderungen. 3 Features mit klar trennbaren Architektur-Mustern:

- **FEAT-651 Theming + Polish:** Tailwind-Layer-Aenderung (tailwind.config.ts) + iterative Migration. Kein neuer Service.
- **FEAT-652 Compliance:** VIES-HTTP-Adapter + DB-Cache-Tabelle (MIG-030) + 1 PDF-Block-Pendant zu V5.7-NL-Variante. Externer HTTP-Call mit graceful-degradation.
- **FEAT-653 Hygiene:** SQL-Bulk-Migration mit User-Mapping-File (MIG-031) + npm-CVE-Akzeptanz-Doku (ISSUE-058). Keine neue Code-Surface.

Architektur-Profil: niedriges Risk, hohe Reuse, kein Architektur-Pivot.

## Data Model / Storage Direction

### MIG-030 — `vat_id_validations` (FEAT-652)

```sql
CREATE TABLE vat_id_validations (
  id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  country VARCHAR(2) NOT NULL,
  number VARCHAR(50) NOT NULL,
  is_valid BOOLEAN NOT NULL,
  vies_response JSONB,
  source TEXT NOT NULL CHECK (source IN ('vies', 'vies_unavailable', 'format_only')),
  validated_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  expires_at TIMESTAMPTZ NOT NULL,
  UNIQUE(country, number)
);
```

- Einzige neue Tabelle in V6.5
- Single-User-Footprint: <100 Rows realistisch
- TTL via `expires_at`, Cache-Pruefung via `WHERE country=$1 AND number=$2 AND expires_at > NOW()`
- Audit-trail via `audit_log` mit action='vies_lookup'

### MIG-031 — Source-Migration ohne Schema-Aenderung (FEAT-653)

- KEINE neue Tabelle, KEIN ALTER COLUMN
- Bulk-UPDATE auf bestehende `contacts.campaign_id` und `companies.campaign_id`
- `contacts.source/source_detail` und `companies.source_type/source_detail` bleiben als read-only-Backup (DEC-160)

## Data Flow / Request Flow

### VIES-Lookup-Flow (FEAT-652)

```
[User editiert VAT-ID in Branding-/Company-Form]
    ↓
[Format-Check synchron via vat-id.ts] → bei invalid: rotes Badge, kein VIES-Call
    ↓ (format-OK)
[Optional: lookupVatId(country, number)]
    ↓
[DB-Cache-Pruefung: SELECT WHERE country=$1 AND number=$2 AND expires_at > NOW()]
    ├─ Hit: Return cached result (gruenes/gelbes Badge je is_valid)
    └─ Miss: VIES-HTTP-Call (~500-2000ms)
        ├─ Success: Insert/Update mit expires_at = NOW() + 24h, audit_log-Insert source='vies'
        ├─ Network-Fail: graceful-degradation, Insert mit source='vies_unavailable', warning-Badge
        └─ VIES-says-invalid: Insert mit is_valid=false, rotes Badge
```

### DE-§13b-Reverse-Charge-Render-Flow (FEAT-652)

```
[User klickt "Angebot erstellen" mit Branding-Country=DE, Company-Country=EU-non-DE, Company-VAT-ID-vorhanden]
    ↓
[Editor zeigt Reverse-Charge-Toggle als enabled (V5.7-3-Voraussetzungen-Logik)]
    ↓ (User aktiviert Toggle)
[saveProposal mit reverse_charge=true]
    ↓
[generateProposalPdf(branding={country:'DE'}, ...)]
    ↓
[proposal-renderer.ts: import DE-block oder NL-block je branding.country]
    ↓
[de-reverse-charge-block.ts liefert pdfmake-Content mit DE-Phrase + § 13b UStG-Verweis]
    ↓
[PDF gerendert, ICP-Equivalent-Inline-Note im Editor sichtbar (kein PDF-Insert)]
```

### Source-Migration-Flow (FEAT-653)

[SLC-656 Step 1: Pre-Migration-Audit-Skript run]

## Security / Privacy Considerations

### DSGVO-Konformitaet

- VIES-Cache speichert keine personenbezogenen Daten (nur VAT-IDs, public registries)
- Source-Migration: Audit-Log-Trail dokumentiert wer wann welche Mapping-Werte verwendete
- Source-Felder als read-only-Backup behalten (DEC-160) — User kann nachvollziehen was wo herkam

### RLS-Pattern

- `vat_id_validations` bekommt RLS-Policy `authenticated_full_access` (Single-User-Internal-Test-Mode-Pattern)
- Keine bestehende RLS-Policy geaendert

### Pre-Production-Compliance-Gate

- **DE-§13b Phrase ist PRELIMINARY** (DEC-162) — Anwaltspruefung Pflicht vor erstem echten Kunden-PDF mit DE-Reverse-Charge
- VIES-Caching: 24h-TTL ist VIES-Standard, keine VIES-Datenschutz-Verletzung
- ISSUE-058 postcss-CVE bleibt offen — Pre-Production-Pflicht: Re-Audit bei Next-Toolchain-Update

### Theming

- Brand-Colors als Tokens machen Color-Konsistenz audit-bar (Cross-File grep auf einzelne Tokens vs. 178 verstreute Hex-Werte)
- Kein Sicherheits-Impact

## Constraints and Tradeoffs

### Theming-Tradeoffs (FEAT-651)

- **Phase-A-only-Erst:** Tokens da, aber Migration nur 2 Files → Drift-Reduktion klein, aber kein Visual-Regress (DEC-152)
- **Iterative Migration:** Mehrere Slices statt Big-Bang → Zeit-Verteilung, kein Sprint-Block (DEC-153)
- **Manual Page-Smokes statt Snapshot-Tests:** Schneller, aber subjektiver. V6.4-Pattern hat sich bewaehrt (DEC-154)
- **PageHeader Slot-Pattern:** Erweitert bestehende Component statt neuer — backward-kompatibel (DEC-156)

### Compliance-Tradeoffs (FEAT-652)

- **DB-Cache statt In-Memory:** survives Container-Restart (Coolify-Redeploy nicht-disruptiv) aber 1 zusaetzliche Tabelle (DEC-157)
- **Graceful-Degradation transparent:** User sieht VIES-Status-Badge, kann trotzdem speichern. Tradeoff: User koennte VIES-Down-State uebersehen → Mitigation: Warning-Badge-Color (gelb statt gruen) (DEC-158)
- **DE-§13b PRELIMINARY:** Liefert Code, aber erfordert Anwalts-Sign-Off vor Customer-Live (DEC-162)

### Hygiene-Tradeoffs (FEAT-653)

- **SQL-Skript statt UI-Tool:** 5-8h gespart, aber weniger user-friendly (DEC-159) — V6.5 ist Hintergrund-Sprint mit hohem User-Skill, akzeptabel
- **Source-Felder als read-only-Backup behalten:** kein DROP COLUMN (DEC-160) — Storage-Kosten-Tradeoff vernachlaessigbar, Rollback-Path-Plus
- **postcss-CVE akzeptiert statt Force-Patch:** keine catastrophic-Downgrade (DEC-161) — bleibt offen bis Upstream
