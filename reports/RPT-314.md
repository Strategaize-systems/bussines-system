### P2: Recursion-Guard — PASS (nach Bug-Fix)

**Erst-Test (vor Fix)**: 3 Pre-Runs im 60s-Fenster + 1 stuck Run wurde ausgefuehrt mit `outcome=success` statt `skipped`. App-Container-Logs zeigten:

```
[recursion-guard] count failed: invalid input syntax for type json
```

**Root-Cause**: `recursion-guard.ts` nutzte `.contains("action_results", [{type, outcome}])`. supabase-js produziert daraus ein postgrest-Query, das von Postgres mit "invalid input syntax for type json" abgelehnt wird. Defensiver Fallback im Code returnt `allowed=true`, deaktiviert Recursion-Guard.

**Fix** (commit `22673d4`):
- `.contains()` ersetzt durch `.select("action_results")` ohne JSONB-Filter, dann JS-Filter im Loop
- Robust gegen postgrest-jsonb-Quirks
- 2 neue Vitest-Cases: `failed/skipped-Runs zaehlen nicht`, `non-Array action_results werden ignoriert`

**Re-Test (post-Deploy)** (run `00000000-...0000a2`):
```json
{ "status": "skipped",
  "action_results": [{
    "type": "update_field",
    "outcome": "skipped",
    "action_index": 0,
    "error_message": "recursion-limit-exceeded (3/3)"
  }]
}
```

`status=skipped`, error_message enthaelt `(count/limit)` — Recursion-Guard greift jetzt korrekt.

## Findings

### High

#### H-01: recursion-guard `.contains()` produziert "invalid input syntax for type json" (FIXED)

**Vor Fix** ([recursion-guard.ts:38-46](cockpit/src/lib/automation/recursion-guard.ts)):
```typescript
const { data, error } = await supabase
  .from("automation_runs")
  .select("action_results")
  .eq("trigger_entity_id", entityId)
  .gte("started_at", since)
  .contains("action_results", [
    { type: "update_field", outcome: "success" },
  ]);
```

App-Container-Logs zeigten konsistent: `[recursion-guard] count failed: invalid input syntax for type json`. Defensiver Fallback im Code returned `allowed=true`, was den Recursion-Guard funktional deaktiviert hat.

**Auswirkung vor Fix**: Endlos-Schleifen bei ueberlappenden update_field-Workflows waeren NICHT verhindert worden. V1-Sicherheitsmechanik (DEC-138 + Risk-Mitigation aus Slice-File) war aktiv im Code, aber inactive im Runtime.

**Fix** (commit `22673d4`, deployed in Image-Tag `22673d4`):
- `.contains()` entfernt
- Stattdessen: alle Runs im Zeitfenster selektieren, in JS auf `type='update_field' AND outcome='success'` filtern
- Robust gegen postgrest-jsonb-Quirks
- 2 neue Vitest-Cases sichern die JS-Filter-Logic ab (failed/skipped zaehlen nicht; non-Array action_results werden ignoriert)

**Re-Test gegen Live-DB**: P2 PASS. Recursion-Guard greift jetzt mit `error_message='recursion-limit-exceeded (3/3)'`.

**Severity-Begruendung High**: Sicherheitsmechanik gegen Endlos-Loops war live unwirksam. update_field-Loops haetten DB-Last produziert + audit_log-Spam + ggf. Stage-Toggle-Loops fuer Workflow-Regeln. Per Deviation Rule 1 direkt gefixt.

### Medium

Keine.

### Low

#### L-01: deals-Tabelle hat keine `owner_id`-Spalte

`assignee-resolver` liest `entity.data.owner_id` und faellt auf `entity.data.created_by` zurueck. Schema (deals) hat aber **nur** `created_by`. V1 Single-User: praktische Auswirkung minimal (`created_by` ist als Fallback gesetzt). V2 Multi-User: dedizierte `owner_id`-Spalte sollte ergaenzt werden. Backlog-Item fuer V7.

#### L-02: Test-Deals ohne `created_by` koennen create_task-Action failen lassen

Beim ersten P1-Test war der Test-Deal mit `created_by=NULL` befuellt. assignee-resolver wirft `AssigneeResolutionError`, der vom Action-Handler in `outcome=failed` uebersetzt wird. **Production-Impact null** — neue Deals bekommen `created_by` durch die createDeal-Server-Action gesetzt. Test-Lerneffekt: pruefen ob Test-Daten realistische Werte haben.
