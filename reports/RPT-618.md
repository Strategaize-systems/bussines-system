## 2. Live-Befund (A-V812-2, gegen Production-Coolify-DB 91.98.20.191)

Container `supabase-db-k9f5pn5upfq7etoefb5ukbcg-075930079644` (zur Laufzeit aufgeloest):

```
SELECT to_regclass('public.ai_cost_ledger')   →  NULL   (Tabelle existiert NICHT)
SELECT to_regclass('public.audit_log')         →  audit_log (existiert)
Tabellen mit Name *cost*/*ledger*/ai_*         →  ai_action_queue, ai_feedback, (email/campaign) — KEIN Cost-Ledger
Spalten mit Name 'tenant_id' (public schema)    →  (keine)
```

**Root cause:** SLC-909, DEC-281 und ARCHITECTURE.md (Phase 2.3 + Data-Model-Block,
"`ai_cost_ledger.tenant_id` existing seit V6.4") basieren auf einer Cross-Repo-Annahme.
`ai_cost_ledger`/`ai_jobs` ist eine **Intelligence-Studio**-Konstruktion (FK
`ai_cost_ledger.job_id → ai_jobs(id)`, siehe `.claude/rules/backend.md`). BS verfolgt
LLM-Cost ausschliesslich in `audit_log.context` als JSON-String `sculptor_cost_usd` (USD)
und nur fuer den **Sculptor-Flow** — die ~15 anderen `queryLLM()`-Caller (cron
meeting-briefing, daily-summary, classifiers, knowledge/search, ki-workspace,
followup-engine, winloss) schreiben keinerlei Cost. Statische Bestaetigung: 0 Treffer fuer
`ai_cost_ledger` in `sql/migrations/*.sql` und in `cockpit/src/**/*.ts`.

**Konsequenz:** MIG-050 (`get_tenant_cost_sums(p_tenant_id UUID)` →
`SUM(cost_eur) FROM ai_cost_ledger WHERE tenant_id=$1`) ist nicht schreibbar — keine
Tabelle, kein `cost_eur`, kein `tenant_id`. Ein Cap auf den vorhandenen Daten wuerde
genau die Runaway-Pfade (Cron-Loops, Multi-Agent) verfehlen, gegen die der Slice schuetzen
soll → False Confidence. Ein echter Pro-Tenant-EUR-Cap braucht zuerst eine
Cost-Recording-Foundation (eigenes Feature).
