# KI-Gatekeeper Research — State-of-the-Art AI Personal Assistants

## Ziel

Recherche aktueller KI-Personal-Assistenten und Architektur-Patterns als Grundlage fuer die KI-Gatekeeper-Architektur (BL-329, V4) und die KI-Kontext-Features in V3.1 (BL-321 bis BL-324, BL-328).

---

## 1. Marktueberblick: Zwei Lager

Der Markt teilt sich in zwei Ansaetze:

### Autonom (AI handelt selbststaendig)
- **Artisan AI (Ava)**: Vollautonomer AI-SDR, schreibt und sendet Cold Outbound, bucht Meetings, handled Einwaende. ~$2.000+/Monat.
- **11x.ai (Alice)**: Autonomer Outbound-Agent, hyper-personalisiert, eigene Prospect-Datenbank. ~$5k-15k/Monat.

### Assist (AI unterstuetzt, Mensch entscheidet)
- **Clay**: Datenanreicherung + Workflow-Builder. 75+ Datenquellen, Waterfall-Enrichment. Mensch designt Workflows und triggered. ~$149-800/Monat.
- **Lindy.ai**: Komposable AI-Assistenten ("Lindies"). Inbox-Triage, Kalender-Management, Daily Briefing. Konfigurierbare Autonomie. ~$49-99/Monat.
- **Relevance AI**: No-Code Agent-Builder. Approval-Steps an beliebiger Stelle einbaubar. ~$19-199/Monat.
- **Microsoft Copilot for Sales**: Suggest-then-Confirm Modell. Email-Summaries, Meeting-Prep aus CRM-Daten, Follow-up-Vorschlaege. Nie autonomes Senden.
- **Google Gemini for Workspace**: Email-Summaries, Reply-Drafts. Weniger CRM-integriert als Copilot.

### Einordnung Strategaize

Unser Freigabe-Prinzip ("KI handelt NIE autonom am Kunden") positioniert uns klar im **Assist-Lager** — das ist korrekt fuer beziehungsgetriebenes B2B. Keiner der AI-SDR-Anbieter ist fuer Inbound-Analyse + CRM-Kontext-Integration optimiert. Das ist unsere Differenzierung.

## 2. Best Practices — Was wir uebernehmen

### 2.1 Suggest-Approve-Execute (Microsoft Copilot Pattern)

**Der Gold-Standard fuer Human-in-the-Loop:**

```
Email kommt rein
  → AI klassifiziert (Deal, Kontakt, Dringlichkeit, Thema)
  → AI reichert Kontext an (Deal-Status, letzte Interaktion, offene Tasks)
  → AI schlaegt Aktion vor (Antwort-Entwurf, Meeting, Task, Wiedervorlage)
  → Mensch prueft in Tagesbriefing oder Echtzeit-Alert
  → Mensch genehmigt / aendert / lehnt ab
  → System fuehrt genehmigte Aktion aus
```

### 2.2 Inbox-Triage (Lindy.ai Pattern)

- Eingehende Emails automatisch kategorisieren, priorisieren, dem richtigen Deal/Kontakt zuordnen
- Dringliches sofort in Aufgabenliste → Rest in Tageseinschaetzung am naechsten Morgen
- Maps direkt auf BL-329 (KI-Gatekeeper)

### 2.3 Context-First (Clay Pattern)

- Kontext aus allen verfuegbaren Quellen BEVOR eine Aktion vorgeschlagen wird
- Bei uns: CRM-Daten (Deal, Kontakt, Firma, Timeline, letzte Aktivitaet) statt externer Provider
- Maps direkt auf BL-323 (Kontext-Intelligenz: alle Formulare KI-vorbefuellt)

### 2.4 Composable Approval Steps (Relevance AI Pattern)

- Human-Checkpoints an beliebiger Stelle im Workflow einbaubar
- Sauberste Architektur fuer Freigabe-Prinzip
- Queue-basiert: AI schlaegt vor → Queue → Mensch entscheidet → System fuehrt aus

### 2.5 Morning Briefing (Lindy + Copilot Pattern)

- Taeglich aggregieren: Ueber-Nacht-Emails, anstehende Meetings mit Deal-Kontext, ueberfaellige Tasks, Deal-Stage-Aenderungen
- Einheitliche Sicht = Differenzierungsmerkmal (Microsoft macht das pro Tool, nicht unified)
- Maps direkt auf BL-328 (Tageseinschaetzung erweitert) und bestehende Mein Tag Seite

### 2.6 Action Cards (Copilot UX Pattern)

- Diskrete, dismiss-bare Karten mit klaren Aktionen (Genehmigen, Bearbeiten, Ablehnen)
- Confidence-Signale zeigen ("Deal schliesst in 5 Tagen, keine Antwort seit 3 Tagen")
- Keyboard-Shortcuts fuer schnelle Batch-Genehmigung (j/k navigieren, a/r genehmigen/ablehnen)

## 3. Was wir NICHT uebernehmen

| Anti-Pattern | Warum nicht |
|---|---|
| Autonomes E-Mail-Senden (Artisan/11x) | Zerstoert Vertrauen in beziehungsgetriebenem B2B |
| Monolithische AI-Persona (Ava/Alice) | Intransparente Black Box. Wir brauchen Transparenz: warum wurde klassifiziert, welcher Deal |
| Externe Datenanreicherung (Clay 75+ Provider) | V1: nur eigene CRM-Daten. Externe Anreicherung ist V5+ Thema |
| SaaS-Pricing-Modell pro Seat | Wir sind self-hosted, keine recurring SaaS-Kosten |

## 4. Technische Architektur-Patterns

### 4.1 MCP (Model Context Protocol — Anthropic)

- Offener Standard fuer AI ↔ Tools/Datenquellen
- Client-Server Modell (JSON-RPC 2.0 ueber stdio oder HTTP+SSE)
- **Relevanz:** Wir koennten Custom MCP-Server fuer Supabase CRM + IMAP bauen
- **Wichtig:** MCP hat KEIN eingebautes Approval-Layer — Human-in-the-Loop muss auf Application-Ebene implementiert werden
- **Verfuegbare Server:** IMAP, Gmail, Google Calendar, PostgreSQL, Filesystem
- **Empfehlung:** MCP als optionaler Integrations-Layer fuer V4+, nicht als V3.1-Requirement

### 4.2 Email-Ingestion

**IMAP-Polling ist die pragmatische Wahl fuer Self-Hosted:**

| Aspekt | Empfehlung |
|---|---|
| Methode | IMAP-Polling alle 2-5 Minuten |
| Library | `imapflow` (Node.js) |
| Parsing | `mailparser` + `email-reply-parser` |
| Deduplizierung | `message_id` + `in_reply_to` in Supabase |
| Auto-Reply-Erkennung | Headers (`Auto-Submitted`), Subject-Pattern, Body-Laenge |

### 4.3 LLM-Klassifikation (Zwei-Pass-Architektur)

**Pass 1 — Regelbasiert (0 LLM-Kosten, ~40% Volume):**
- Auto-Replies (Header-Check)
- Newsletter (`List-Unsubscribe` Header)
- Bounces (`mailer-daemon`)

**Pass 2 — Claude Sonnet via Bedrock (~$0.003/Email):**
- Input: Subject + erste 500 Zeichen + Sender-Metadata
- Output: `{category, urgency, intent, referenced_entity_hints, summary}`
- Entity-Linking: Domain aus Sender-Email → Firma matchen, Fuzzy-Match auf Kontakte, Top-3 Deals als Kontext

**Batch-Verarbeitung:** Micro-Batches pro Poll-Zyklus (alle 5 Min). Email ist asynchron — kein Real-Time noetig.

### 4.4 Follow-Up-Intelligenz (Hybrid)

**Regelbasiert + LLM:**

```
follow_up_rules Tabelle:
  deal_stage → default_interval
  - Hot Lead:       2 Tage
  - Proposal Sent:  3 Tage
  - Warm:           5 Tage
  - Cold:           7 Tage
  (konfigurierbar pro Deal)

State Machine pro Thread:
  awaiting_response → response_received | overdue
  
Smart Rescheduling:
  Auto-Reply "back on Monday" → parse return_date → follow_up = return_date + 1
```

### 4.5 Human-in-the-Loop Queue

**Supabase-Tabelle `ai_action_queue`:**

| Spalte | Typ | Beschreibung |
|---|---|---|
| id | uuid | Primary Key |
| type | enum | reply, follow_up, task, meeting, alert |
| email_id | uuid | Referenz auf analysierte Email |
| deal_id | uuid | Zugeordneter Deal |
| suggested_action | jsonb | AI-Vorschlag (Draft, Datum, etc.) |
| ai_reasoning | text | Begruendung der AI |
| status | enum | pending, approved, rejected, expired |
| created_at | timestamptz | Erstellt |
| decided_at | timestamptz | Entschieden |

**Morning Review UX:** Einzelne Seite, sortiert nach Dringlichkeit. Pro Karte: Email-Summary, AI-Vorschlag, Approve/Reject/Edit Buttons. Keyboard-Shortcuts. Ziel: 10 Entscheidungen in 2 Minuten.

**Eskalation:** pg_cron prueft `pending` Items aelter als 4 Stunden waehrend Geschaeftszeiten → Push-Notification.

### 4.6 DSGVO / Compliance

| Aspekt | Loesung |
|---|---|
| Datenresidenz | Bedrock Frankfurt (eu-central-1) + Hetzner Deutschland = alles in EU |
| Email-Speicherung | Volle Email 90 Tage, danach nur Metadata + AI-Summary (kein Original-Body) |
| Rechtsgrundlage | Berechtigtes Interesse Art. 6 Abs. 1 lit. f DSGVO |
| Audit Trail | `ai_audit_log` Tabelle: jede Klassifikation, jeder Vorschlag, jede Entscheidung. Insert-only. |
| Art. 22 DSGVO | Recht auf Erklaerung automatisierter Entscheidungen → ai_reasoning Feld |

## 5. Architektur-Empfehlung fuer Strategaize

### Phase 1: V3.1 — Kontext-Intelligenz (ohne IMAP)

Kein IMAP noetig. Nutzt nur vorhandene CRM-Daten:

- **BL-323 Kontext-Intelligenz:** Formulare aus Deal/Kontakt/Firma-Kontext vorbefuellen
- **BL-322 KI-E-Mail-Composing:** Voice + KI-Verbesserung, Kontext-Vorbefuellung
- **BL-328 Tageseinschaetzung erweitert:** Rueckblick Vortag aus vorhandenen CRM-Events
- **BL-321 Pipeline KI-Suche:** Natuerlichsprachliche Suche via Bedrock
- **BL-324 Auto-Wiedervorlagen:** Regelbasiert nach Meeting/Call/Email

**Tech:** Bestehender Bedrock LLM-Service (FEAT-305) + neue API-Routen + Queue-Tabelle.

### Phase 2: V4 — KI-Gatekeeper (IMAP erforderlich)

IMAP-Integration als Voraussetzung:

- **FEAT-405 IMAP Mail-Integration:** Email-Ingestion, Parsing, Speicherung
- **BL-329 KI-Gatekeeper:** Klassifikation, Priorisierung, Zuordnung, Alerting
- **BL-325 KI-Wiedervorlagen:** Bidirektional Mensch ↔ KI
- **BL-327 Auto-Reply-Erkennung:** Intelligentes Rescheduling

**Tech:** `imapflow` + Zwei-Pass-Klassifikation + `ai_action_queue` + Morning Review UI.

### Phase 3: V4+ — MCP Integration (optional)

- Custom MCP-Server fuer Supabase CRM-Daten
- Custom MCP-Server fuer IMAP
- Ermoeglicht Claude Desktop / externe Agents als Frontends
- Nur sinnvoll wenn Multi-Client-Zugriff auf CRM-Daten gebraucht wird

## 6. Differenzierung gegenueber Wettbewerb

| Aspekt | Markt | Strategaize |
|---|---|---|
| Fokus | Outbound-Sales | Inbound-Analyse + Beziehungspflege |
| Autonomie | Autonom oder konfigurierbar | Freigabe-Prinzip: nie autonom am Kunden |
| Daten | Externe Provider (Clay: 75+) | Eigene CRM-Daten als Single Source of Truth |
| Hosting | SaaS (US-Cloud) | Self-Hosted Hetzner Deutschland (DSGVO) |
| Voice | Kaum vorhanden | Whisper-first fuer Mobile + Desktop |
| Preis | $49-$15.000/Monat recurring | Fixe Infrastrukturkosten, keine Seat-Lizenz |
| Integration | Generisch | Tief in eigenes CRM integriert (Deal, Kontakt, Firma, Timeline) |

---

## 7. Fazit

Die Recherche bestaetigt:

1. **Freigabe-Prinzip ist korrekt** — der Markt zeigt, dass autonomes Handeln bei B2B-Beziehungen Vertrauen zerstoert
2. **Suggest-Approve-Execute ist der Gold-Standard** — Microsoft Copilot hat das Pattern validiert
3. **Morning Briefing als zentrale UX** — alle erfolgreichen Assistenten bieten einen Daily Summary
4. **Zwei-Pass-Klassifikation** spart ~40% LLM-Kosten durch regelbasierte Vorfilterung
5. **Queue-basiertes Approval** ist die sauberste Architektur fuer Human-in-the-Loop
6. **V3.1 braucht kein IMAP** — Kontext-Intelligenz aus vorhandenen CRM-Daten reicht
7. **MCP ist nice-to-have fuer V4+**, nicht kritisch fuer die Kernfunktionalitaet
8. **Voice-Input ist echtes Differenzierungsmerkmal** — kein Wettbewerber hat das als Kernfeature
