## 6. Problems Found

1. **Slice-Spec sent_at-Inkonsistenz (Auto-Fix in Implementation):** Scope-Section sagte "kein Timestamp-Update bei sent" waehrend AC3 + Confirm-Modal-Text "Datum=heute" das Setzen von `sent_at` implizierten. Pragmatisch geloest durch Setzen von `sent_at = now()` bei sent-Transition (semantisch korrekt, Spalte existiert genau dafuer in V2). In IMP-231 (Dev-System SKILL_IMPROVEMENTS) als Pattern-Vermerk dokumentiert: Lifecycle-Slice-Specs brauchen zentrale Status-zu-Timestamp-Mapping-Tabelle.
2. **Slice-Spec Cron-Auth-Header-Drift (Auto-Fix in Implementation + REL-020-Notes):** Slice-Spec MT-3 + MT-8 sagten `Authorization: Bearer + CRON_SECRET`, der existierende Pattern in `verify-cron-secret.ts` nutzt aber `x-cron-secret`. Der Spec-Wortlaut waere zu 401 beim ersten Cron-Smoke gefuehrt. REL-020-Notes + cURL-Smoke + `node -e fetch(...)`-Command auf `x-cron-secret` umgestellt. In IMP-230 (Dev-System) dokumentiert: /architecture muss bei neuen Cron-Endpoints den existing Helper-Pattern explizit referenzieren statt generischen Standard.
3. **TypeScript-Inferenz-Drift bei `getProposalVersionsChain`:** Initial-Implementation mit `let rootId: string | null` und `const { data } = await supabase...maybeSingle()` triggerte Self-referential-Type-Error (TS2615). Auto-Fix mit explicit-Typing der Lookup-Response (`{ data: { id, parent_proposal_id } | null }`) — Build PASS nach Fix.
4. **Coolify-Cron-Setup ist User-Action-Out-of-Scope:** Der Cron-Endpoint ist im Code + Build, aber der Coolify-Cron-Job muss nach Deploy manuell vom User angelegt werden (laut feedback_manual_deploy.md). REL-020-Notes haben die Anleitung. Pre-Deploy-Status: Auto-Expire faellt aus bis Cron-Anlage. Sicherheitsneutral (manuelle Status-Transitions funktionieren).
