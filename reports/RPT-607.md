# V8.12 Defense-in-Depth Sprint — Requirements

## 1. Problem Statement

V8.11 hat die DB-Layer RLS-Architektur 100% scharf gemacht (Q-V8.11-B Done-Gate=0 persistent). Dabei sind **bewusst 7 Code-Layer-Defense-Gaps** akzeptiert worden — Server-Actions umgehen die DB-RLS via `createAdminClient` ohne is_admin()-Pre-Check. Zusaetzlich wurden **4 Cross-Repo-Security-Audit-Pattern-Slots** (CSP, Passwort-Policy, Logger-Redaction, LLM-Cost-Cap) als "NICE-TO-HAVE vor Customer-Live" deferred + **1 Observability-Gap** (Sentry/Error-Monitoring nicht installiert).

Internal-Test-Mode (Founder allein) macht keine dieser Luecken zum Funktional-Blocker. **Aber bevor irgendein zweiter realer User dazukommt**, muessen die Code-Layer-Bypasses geschlossen sein (sonst sind sie als Pattern in der Codebasis und werden kopiert) und Cross-Repo-Polish-Patterns auf BS-Seite genauso scharf sein wie sie auf ImSch + IS bereits sind.

## 2. Goal / Intended Outcome

**Code-Layer-Defense-in-Depth so eng wie die DB-Layer-RLS jetzt schon ist** + **Cross-Repo-Hygiene-Patterns auf BS-Seite einrichten** + **Observability-Baseline (Sentry) live**.

Konkret nach V8.12 STABLE:
- 0 Server-Actions umgehen DB-RLS via `createAdminClient` ohne is_admin()-Pre-Check
- BS hat CSP-Headers im Production-Mode (strict nach Report-Only-Phase)
- BS Passwort-Policy 12+ Zeichen + zxcvbn-Score >=3 fuer neue Passwoerter
- BS hat Logger-Redaction-Layer fuer Security-Keys + Minimal-PII
- BS hat LLM-Cost-Cap (Pro-Tenant DAILY+MONTHLY) im Bedrock-Adapter
- BS hat Sentry-Error-Monitoring (EU-Region Frankfurt) mit CSP-report-uri-Integration

## 3. Primary Users / Operators

- **Founder** (Internal-Test-Mode, alleiniger User bis Modul 1+2+3 komplett — siehe `module-lifecycle-discipline.md`)
- **System-Operatoren** (Sentry-Telemetry, Cost-Cap-Alerting, Logger-Inspektion via Coolify)

Keine Customer-User in V8.12-Scope. Pre-Customer-Live-Slot fuer Passwort-Force-Reset bleibt separater Pre-Live-Slot (siehe DEC-278).

## 8. Risks / Assumptions

### Risiken

- **R-V812-1 (Phase 1)**: Code-Layer-Switch-Pattern koennte UI-Regressionen verursachen, wenn z.B. UI ein Admin-only-Feature an Member zeigt das vorher via createAdminClient lief. Mitigation: Live-Smoke 7 UI-Pfade.
- **R-V812-2 (Phase 2 BL-501)**: CSP strict-Phase koennte 80+ Pages brechen, wenn Inline-Scripts/Styles uebersehen. Mitigation: Phase-A Report-Only 1-2 Wochen + Iterativ-Fix.
- **R-V812-3 (Phase 2 BL-502)**: zxcvbn-Bundle-Size ~700KB minified — koennte App-Build-Size signifikant erhoehen. Mitigation: dynamic import in set-password/accept-invitation Routes.
- **R-V812-4 (Phase 2 BL-504)**: Pre-flight-Check fuegt DB-Query-Latenz vor jedem Bedrock-Call. Mitigation: Cache fuer 1min via In-Memory + cache-bypass bei Cap-Approach.
- **R-V812-5 (Phase 3 FEAT-923)**: Sentry-DSGVO-Posture — User-IPs werden default eingesammelt. Mitigation: `Sentry.init({ sendDefaultPii: false })` + Custom-Event-Filter via beforeSend-Hook.
- **R-V812-6 (Phase 3 FEAT-923)**: Sentry-Cost koennte bei Cap-Hit spike (`Error-Volume × Sentry-Event-Cost`). Mitigation: SENTRY_TRACES_SAMPLE_RATE=0.1 + Rate-Limit pro Issue-Type.

### Assumptions

- **A-V812-1**: `assertAdmin()` Helper-Function existiert oder ist trivial via `is_admin()` Helper-RPC kombinierbar. Falls nicht: 1 neue Helper-Function in Phase-1 vor allen 7 Closures.
- **A-V812-2**: ai_cost_ledger.tenant_id ist konsistent gepflegt fuer alle Bedrock-Calls. Falls nicht: BL-504 muss Default-Tenant-Fallback haben.
- **A-V812-3**: Sentry.io EU-Region Free-Tier reicht fuer Internal-Test-Mode Burn-In. Upgrade auf Team-Plan erst wenn Volume gross wird.
- **A-V812-4**: Cross-Repo Logger-Redaction Pattern in ImSch oder IS existiert noch nicht — BS V8.12 ist Pattern-Origin. Falls existing: 1:1 portieren.
