### Bedrock-Prompt-Template (DEC-220)

- **System**: "Du bist Vertriebs-Analyst eines B2B-Beratungsunternehmens. Auf Basis der Activity-History eines Deals sollst du den wahrscheinlichsten Verlustgrund vorschlagen. Antworte ausschliesslich auf Deutsch und ausschliesslich als JSON."
- **Input**: Deal-Meta + letzte 10 Activities + letzte 3 E-Mail-Threads (Subject + Snippet[0..200])
- **Output-Schema**: `{ "suggestions": [{ "reason": "...", "source": "..." }, ...] }`
- **Token-Budget**: ~500 Input + ~300 Output, Cost ~$0.005-0.01 pro Call
- **Fallback**: Bei Parse-Error / Bedrock-Error / Empty-History → Modal oeffnet trotzdem mit leerem Feld + Info-Hint

### Data Flow (Stage-Move mit Pflichtfeldern + KI-Suggest fuer Verloren)

```
Drag-Drop "Verloren" 
  → Client prueft STAGE_REQUIRED_FIELDS["Verloren"] 
  → Client-Call suggestLossReason(dealId)
     → Server: deal + 10 activities + 3 emails → Bedrock Sonnet → JSON parse → audit_log insert
     → Return { primary, alternatives, costUsd }
  → Modal oeffnet pre-filled
  → User confirms 
  → Client-Call moveDealToStage(dealId, newStageId, "Verloren", { won_lost_reason: "..." })
     → Server: UPDATE deals SET won_lost_reason + UPDATE deals SET stage_id + audit_log + automation trigger
  → revalidatePath + Toast
```

### Tradeoffs

- **Modal generic fuer alle 5 Stages, KI-Suggest nur Verloren**: + konsistente UX, − Modal-Hint noetig fuer 4 Stages ohne KI-Suggest
- **Keine neue Schema-Migration**: + schmaler V8-Scope, − Accepted/Edited/Rejected-Tracking fuer KI-Suggest deferred V9
- **String-Substitution statt Mass-Rename**: + V8 bleibt schmal, − Mental Mismatch Code-Naming (bedrock-client.ts) vs UI-Naming (KI)
- **Goals-Move statt -Delete**: + erhaelt Feature, − eine Datei-Verschiebung mehr in SLC-811
