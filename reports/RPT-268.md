## Befund + Fix pro Item

### Item 1 â€” ISSUE-047 Hydration #418

**Code-Audit-Befund:** Die ISSUE-047-Vermutung "Datums-Format-Drift im Listing-Card" stimmt **nicht** mit dem Code-Stand ueberein:

- `cockpit/src/app/(app)/proposals/proposals-client.tsx` rendert kein Datum
- Grep ueber `(app)/proposals` nach `toLocale|formatDate|new Date()|Math.random|Date.now`: keine Treffer im Render-Pfad
- `cockpit/src/components/proposal/proposal-status-badge.tsx`: kein Datum
- Sidebar + Layout-Tree (`(app)/layout.tsx` + Root-Layout): kein Datum, kein non-deterministischer Render

**Wahrscheinliche Ursache:** Browser-Extension am `<body>` (Pattern: dark-mode-extension, password-manager-injects, translation-extensions). Diese modifizieren das DOM nach dem Server-Render und produzieren den #418-Mismatch ohne dass der App-Code beteiligt ist.

**Fix:** `suppressHydrationWarning` auf `<html>` + `<body>` im Root-Layout. Standard-Pattern fuer extension-induzierten Drift, unterdrueckt **nur** den Top-Level-Diff, nicht Children-Komponenten.

**Files changed:**
- `cockpit/src/app/layout.tsx` (4 Zeilen)

**Limitation:** Falls #418 weiterhin auftritt, ist die Quelle tiefer (Provider-Race, Auth-State-Drift, dynamic content). Dann braucht es Live-Console-Inspection im Browser.

### Item 2 â€” SLC-541 M1 ConditionalColorPicker derived-state

**Befund:** Komponente nutzte `useState` + `useEffect` um `enabled` aus `value` abzuleiten. Das fuehrt zu Sync-Drift bei Eltern-Wechsel (useEffect feuert nach Render, intermediate state inkonsistent).

**Fix:** `useState`/`useEffect` entfernt, `enabled = value !== null` als pure derived state. Eltern-State (Form) ist Single-Source-of-Truth. Toggle-Click ruft `onChange(checked ? defaultColor : null)` auf â€” das setzt value, daraus folgt enabled.

**Files changed:**
- `cockpit/src/components/branding/conditional-color-picker.tsx` (komplett umgeschrieben â€” 87 â†’ 75 Zeilen, useEffect-Block weg)

**Behavior unchanged:** AC9-Drift-Fix (FEAT-531) bleibt aktiv â€” User mit `primary_color = NULL` sieht Toggle aus, Mail rendert weiter ueber `textToHtml`-Fallback.

### Item 3 + 4 â€” Total-Size-Limit + Filename-Kollision (gemeinsame Server-Side-Aenderung)

**Befund (ISSUE-045):** `validateAttachment` wurde mit `totalSizeSoFar = 0` aufgerufen, das 25-MB-Total-Limit war reine Browser-Convenience. Client-Bypass via direkter API-Call konnte beliebig viele 10-MB-Files in derselben Compose-Session hochladen.

**Befund (SLC-542 L1):** Upload nutzte `upsert: true` â€” bei Filename-Kollision (z.B. User dragged "report.pdf" zweimal aus verschiedenen Quellen) wurde die alte Datei still ueberschrieben. Re-Drag-Idempotenz war erwuenscht, echte Kollision wurde aber nicht erkannt.

**Gemeinsamer Fix in `/api/emails/attachments` POST:** Single `list()`-Call deckt beide Aspekte ab:

```ts
const { data: existingFiles } = await admin.storage
  .from(BUCKET)
  .list(`${user.id}/${composeSessionId}/`, { limit: 1000 });
const existing = existingFiles ?? [];

// (a) ISSUE-045 â€” Total-Size aus Storage-State berechnen
const totalSizeSoFar = existing.reduce((sum, f) => sum + (f.metadata?.size ?? 0), 0);

// (b) SLC-542 L1 â€” Filename-Kollision-Suffix
const safeName = sanitizeFilename(file.name);
const existingNames = new Set(existing.map((f) => f.name));
const finalName = generateUniqueName(safeName, existingNames);
```

`generateUniqueName` haengt " (n)" vor die Extension wenn Kollision vorliegt: `report.pdf` â†’ `report (1).pdf` â†’ `report (2).pdf` â†’ ...

`upsert: true` â†’ `upsert: false` fuer den Storage-Upload. Re-Drag derselben Session erzeugt jetzt Suffix-File statt blind zu ueberschreiben.

**Files changed:**
- `cockpit/src/app/api/emails/attachments/route.ts` (~30 Zeilen)

**Performance:** 1 zusaetzlicher list()-Call pro Upload (~50-100ms). Bei 25-MB-Sessions und Pro-Upload-Latenz von mehreren Sekunden vernachlaessigbar.
