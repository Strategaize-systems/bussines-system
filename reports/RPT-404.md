## 4. HIGH Finding — ISSUE-065

### Befund

`cockpit/src/lib/team/aggregate-queries.ts` `getTeamMembers()` Zeile 188-192:

```typescript
const { data: callerProfile } = await supabase
  .from("profiles")
  .select("id, team_id")
  .single();
```

Diese Query nutzt `.single()` OHNE einen `.eq("id", auth.uid())`-Filter. Die `profiles_select_team` RLS-Policy liefert dem Teamlead 6 Rows (`team_id = get_my_team_id()` ODER `id = auth.uid()`). `.single()` returnt bei multi-row Result `null`, daher:

```typescript
const callerTeamId = caller?.team_id ?? null;  // -> null
const callerId = caller?.id ?? null;            // -> null

if (callerTeamId) { /* nicht angewendet */ }
if (callerId)    { /* nicht angewendet */ }
```

**Effekt:** Self-Filter ist im Code vorhanden aber inaktiv. Tabelle zeigt 6 Rows statt 5 (Teamlead-self included). `getTeamBedrockContext()` ruft `getTeamMembers()` → gleicher Drift in Bedrock-Coaching-Prompts.

### Verifikation

- Live-Page: 6 Tabellen-Rows ([TEST] Test-Member 1..5 + [TEST] Test-Teamlead)
- Bedrock-Antwort: erkennt Teamlead als Outlier ("Pipeline 0 €, 0 offene Aktivitaeten"), gibt selbst Klarstellung "Teamlead-Rolle typischerweise nicht auf eigene Pipeline ausgelegt"

### Severity & Impact

- **HIGH** (nicht Blocker): funktional kein Daten-Leak, UX-Drift + Bedrock-Coaching-Drift
- Test-Coverage hat den Bug verfehlt: `aggregate-queries.test.ts` repliziert die SQL-Logik manuell mit `id != caller.id`, ruft NICHT `getTeamMembers()` direkt auf. Dieser Test-Tradeoff war in RPT-403 bewusst dokumentiert ("Production-Code muessen daher synchron gehalten werden — TODO").

### Empfohlener Fix

```typescript
const { data: { user } } = await supabase.auth.getUser();
if (!user) return [];

const { data: callerProfile } = await supabase
  .from("profiles")
  .select("id, team_id")
  .eq("id", user.id)
  .single();
```

Oder via SQL-RPC mit `auth.uid()` server-side. Aufwand: 1 MT, ~30min. Empfehlung: als Pre-SLC-706 Hotfix in eigenem Commit.
