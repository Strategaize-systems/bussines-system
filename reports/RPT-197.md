## Outcome summary

**PASS — voll funktional.** SLC-514 ist abgeschlossen und produktionsreif. Ein realer Test-Call (Browser → Asterisk Echo-Extension 600, 42 Sekunden Audio) wurde komplett durch die Pipeline verarbeitet: Upload → Whisper-Transkript → Bedrock-Summary → Deal-Timeline-Activity.

## Findings (alle resolved)

### ISSUE-039 — Recording-Volume fuer nextjs-User nicht lesbar (Blocker, resolved)
- `drwxr-x---` (0750, UID 101/asterisk) vs. app-Container `nextjs` (UID 1001)
- Fix: `asterisk/entrypoint.sh` ergaenzt um `chmod 0755 /var/spool/asterisk/monitor/` + `umask 022`
- Commit `d0bf898`, verifiziert via Volume-`ls -ld` post-redeploy: `drwxr-xr-x`

### Codec-Issue (Blocker, resolved)
- `codec_opus.so` nicht im Ubuntu-asterisk-Paket bundled
- Browser sendet Opus (WebRTC-Standard), Asterisk kann nicht uebersetzen → 40+ "No translator path" Warnings pro Call, MixMonitor schreibt 44-Byte-Header-only-WAVs
- Fix: Opus aus PJSIP `allow` entfernt, nur `ulaw`. Browser + SIP.js verhandeln dann PCMU im SDP-Answer.
- Commit `ccbe6e2` (gemeinsam mit direct_media-Fix)

### direct_media = true (Blocker, resolved)
- Asterisk-Default route RTP peer-to-peer, Asterisk sah kein Audio → Echo() + MixMonitor leer
- Fix: `direct_media = no` explizit am webrtc-user-Endpoint
- Commit `ccbe6e2`

### Dialplan Extension 600 hatte kein MixMonitor (Medium, resolved)
- Extension 600 rief nur Answer+Playback+Echo ohne Aufzeichnung
- Fix: MixMonitor mit X-Call-ID-Header analog zu 1001 hinzugefuegt
- Commit `736453b`

### ISSUE-040 — Supabase Storage Uploads broken, latent seit letztem Upgrade (Blocker, resolved)
- `supabase_storage_admin` konnte `set_config('role','service_role',...)` nicht ausfuehren → PostgreSQL-Error 42501, von storage-api als "new row violates row-level security policy" gemeldet
- Zusatz: fehlende `GRANT USAGE ON SCHEMA storage` + kein `search_path=storage,public` fuer service_role/anon/authenticated
- **Latent auch fuer Meeting-Recordings** (V4.1) kaputt — bisher nicht bemerkt, da seit Upgrade kein Meeting-Recording stattgefunden hat. Storage `meeting-recordings` + `call-recordings` Buckets waren objektleer.
- Fix: MIG-021 — Role-Membership + Schema-Grants + search_path. Zusaetzlich wurden die in MIG-020 angelegten fehlerhaften `call_recordings_*` RLS-Policies entfernt (BYPASSRLS via service_role ist der richtige Weg, genauso wie meeting-recordings es seit V4.1 tut)
- Live-Patch + Migration-Datei `021_v51_storage_grants_fix.sql`

### Medium aus RPT-196 nachgezogen

- Buffer-Size-Check auf `< 1000` Byte angehoben (vorher: `< 44`, hat 44-Byte-leere-WAVs durchgelassen). Commit `d0bf898`.

## Observations / Follow-ups (nicht blockend)

- **Audio-Qualitaet ulaw (G.711 8kHz)**: Ausreichend fuer Whisper-Transkription, aber schlechter als Opus. Fuer Production mit hohem Volumen sollte `codec_opus.so` nachinstalliert werden (aus Third-Party-Repo oder Custom-Build). Separater IMP-Kandidat.
- **SIP_TRUNK_ENABLED=false**: Aktuell keine ausgehenden Anrufe an externe Nummern moeglich. Fuer Go-Live muss ein SIP-Trunk-Provider (Sipgate, Telekom etc.) konfiguriert werden. Dokumentiert als Go-Live-Voraussetzung.
- **Playback demo-echotest failed**: Cosmetischer Warning in Asterisk — die englische Echo-Ansage spielt nicht ab (wahrscheinlich Codec-Mismatch bei der Standard-Sound-Datei), aber Echo() selbst funktioniert perfekt. Kein Impact auf SLC-514.
- **DSGVO: TRANSCRIPTION_PROVIDER=openai**: Whisper laeuft aktuell ueber OpenAI-API-US. Laut data-residency.md sollte stattdessen Azure OpenAI EU genutzt werden. Bestehender Zustand seit V4.1, nicht durch SLC-514 eingefuehrt — separates IMP.
