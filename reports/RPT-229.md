## Findings

### Medium

#### F-1: Stage-Mapping-Drift gegen Live-Pipeline-Stages (waehrend QA gefixt)

**File:** `cockpit/src/app/(app)/emails/compose/recipient-suggest.ts:21-37` (vor Fix)

**Befund:** Die hartkodierte `STAGE_TO_SUBJECT`-Map enthielt nur Slug-IDs (`discovery`, `proposal`, `closed_won`) und einige DE-Schlagworte (`erstkontakt`, `gewonnen`). Die Live-Pipeline hat aber 22 frei formulierte deutsche Stages (z.B. `"Erstkontakt erfolgt"`, `"Verhandlung / Einwaende"`, `"Strategischer Multiplikator"`, `"Vertiefung / Bedarfsschaerfung"`). Der `subjectForStage`-Lookup nutzte exact-match nach `toLowerCase + replace whitespace`, also `"Erstkontakt erfolgt"` → `erstkontakt_erfolgt` → kein Match → return null.

**Impact:** AC4-Subject-Fuellung schlug bei allen 4 aktuell vorhandenen Live-Test-Deals fehl. `to` wurde aus Inbound-Mail korrekt gefuellt, aber `subject` blieb leer. AC4 sagt explizit: "subject (Stage-basiert)" — Implementation entsprach dem nicht.

**Severity:** Medium (funktional partiell — `to`-Fuellung funktionierte).

**Fix (Deviation Rule 1 — Bug-Fix, einfache Aenderung):** Ersetzt durch Substring-Match-Logik mit 11 Regex-Regeln in priorisierter Reihenfolge (Closing > Verhandlung > Angebot > Gespraech > Top-of-funnel). Manuelle Verifikation gegen alle 22 Live-Stages: jede produziert jetzt einen sinnvollen Subject-Vorschlag.

```diff
- const STAGE_TO_SUBJECT: Record<string, string> = { ... 12 Eintraege ... };
- const key = stage.toLowerCase().replace(/\s+/g, "_");
- return STAGE_TO_SUBJECT[key] ?? null;
+ const STAGE_RULES: StageRule[] = [
+   { match: /\b(gewonnen|won|closed_won)\b/i, subject: "Vielen Dank fuer dein Vertrauen" },
+   { match: /\b(verloren|lost|disqualifiziert)\b/i, subject: "Kurzes Update von uns" },
+   ... 9 weitere Regeln ...
+ ];
+ for (const rule of STAGE_RULES) if (rule.match.test(stage.toLowerCase())) return rule.subject;
+ return null;
```

**Status:** Fixed in QA-Run. tsc + Vitest gruen. Fix-Commit + Re-Deploy noetig (User-Action, gleiche Coolify-Pipeline wie SLC-533-Deploy).

### Low

#### F-2: Mobile-Tab-Switch resettet lokalen Component-State (Spec-konform, Risk dokumentiert)

**File:** `cockpit/src/app/(app)/emails/compose/compose-studio.tsx:187-189`

**Befund:** Mobile-Tabs nutzen conditional rendering (`{mobileTab === "templates" && templatesPanel}`). Wenn der User auf Mobile zwischen Tabs wechselt, werden die inaktiven Panels **unmounted**. Lokaler Component-State geht verloren:
- TemplatesPanel: `filter` (System/Eigene/Alle) und `category` (Dropdown) resetten zu Default
- ComposeForm: `improving`, `improveResult`, `improveError`, `suggestNote` (UI-Bubbles) resetten

Container-State (`to`, `subject`, `body`, `templateId`, `language`, `followUp`, `mobileTab`) ueberlebt — das ist die Spec-Mitigation, die vorgesehen ist.

**Impact:** Mobile-only-UX-Wart. Beispiel: User filtert Templates auf "Eigene", wechselt zu "Erfassen", wechselt zurueck → Filter ist wieder "Alle". Auf Desktop kein Problem (alle 3 Panels immer mounted).

**Severity:** Low. Slice-Spec dokumentiert dieses Verhalten als Risk und akzeptiert es als Mitigation: "State liegt im Eltern-Container, Tabs rendern nur Inhalte conditionally".

**Fix-Option (NICHT in QA umgesetzt):** CSS-basiertes Hide-statt-Unmount (`hidden`-Class statt conditional rendering). Wuerde alle 3 Panels permanent mounten und State persistieren.

**Empfehlung:** SLC-535 Polish-Task — wenn der Mobile-only-Use-Case in Praxis relevant wird (heute: Mein-Tag/Focus arbeitet vorwiegend Desktop). Aktuell akzeptiert.
