## 6 Problems Found

**Keine Blocker, keine offenen Inkonsistenzen.** Folgende kleinere Auffaelligkeiten / Vorab-Klaerungen waeren hilfreich, blockieren aber /slice-planning nicht:

- **`useMediaQuery`-Hook-Existenz nicht verifiziert.** Architecture annimmt einen `useSyncExternalStore`-basierten `useMediaQuery`-Hook (analog `feedback_react19_use_mounted_pattern`). Falls BS noch keinen hat, muss /frontend SLC-881 ihn als Pure-Hook-Helper neu erstellen (~30 Min) — kein Blocker, aber als MT-Item planbar.
- **Subagent-Pattern-Reuse-Search hat package.json gelesen statt actual import.** Subagent berichtete BS nutze `@radix-ui/react-dialog`, eine direkte Lese der `dialog.tsx` zeigt aber `@base-ui/react/dialog` (Base UI von Vercel, nicht Radix). Architecture nutzt korrekt Base UI. Lehre: bei Library-Vergleichen lieber den realen Import lesen als nur package.json (Memory-Kandidat fuer Skill-Improvement).
- **Hotspot-Mindest-Anzahl fuer Pilot ist noch offen.** FEAT-881 fordert "min 3 Hotspots" aber keine Maximalzahl. Architecture nimmt 3-5 als praktische V1-Range an (zod-Limit ist 20). Konkrete Anzahl + Body-Text-Authoring gehoert in /slice-planning.
- **Playwright-Capture-Konkretisierung deferred.** Architecture nennt die bestehende `deliverables/user-guide/screencaps.spec.ts` Pipeline, aber konkrete Capture-Konfiguration (welcher Login-State, welche Test-Daten in `mein-tag`, welcher Viewport) gehoert in /slice-planning.
- **`.help-content` CSS aus V8.3 deckt Modal-Body-Render ab?** Modal-Body wird via `dangerouslySetInnerHTML` gerendert. Falls die Modal-CSS-Class anders als `.help-content` heisst, koennte das Modal-Body-Markdown anders aussehen als die Hotspot-Image-begleitende Markdown-Liste. /frontend muss pruefen ob beide ueber `.help-content` laufen oder ob das Modal eine eigene Body-CSS-Schicht braucht (z.B. `.hotspot-modal-content`).
