## Findings (Live-Smoke)

### Medium — ISSUE-048 / BL-418: PaymentTermsDropdown initial Display-Bug

Bei initial render zeigt das base-ui Select-Trigger den Raw-Value `__custom__` statt des Labels `(eigene Eingabe)`. Im Editor sichtbar als `combobox value="__custom__"` und in Mobile-Screenshot als geschriebener Text im Trigger-Button.

**Ursache:** `useState<string>(CUSTOM_VALUE)` setzt den initialen Value als `"__custom__"`. base-ui's `<SelectValue placeholder="(eigene Eingabe)" />` greift den Placeholder nur bei `value=null` oder leerem String — nicht bei einem konkreten Value, dessen SelectItem-children nicht automatisch in den Trigger gerendert werden.

**Funktional intakt:** sobald User klickt + auswaehlt, ist alles korrekt. Cosmetic-Bug, nicht release-blockierend.

**Fix-Optionen:** 
- (A) `useState<string>('')` als Default + onValueChange handelt Empty-String wie CUSTOM_VALUE
- (B) Explicit-render-callback im SelectValue mit Lookup auf templates+CUSTOM_VALUE
- (C) Display-Override: `<SelectValue><span>{currentLabel || '(eigene Eingabe)'}</span></SelectValue>`

Aufwand: ~10min. Backlog: BL-418, KNOWN_ISSUES: ISSUE-048.

### Low — ISSUE-049 / BL-419: UI-State-Drift nach Auto-Save-Error

Nach mehreren Server-rejected Auto-Saves im SkontoSection (Beispiel: Prozent=10 ausserhalb Range, Server validateSkonto rejected) flippt der Toggle visuell in OFF-Zustand. DB haelt aber den vorher erfolgreich persistierten gueltigen State. Selbstheilend nach Page-Reload.

**Beobachtung im Run:**
1. Toggle-On → DB persistiert {2.0, 7}
2. Prozent → 10 → Server reject, Save="Fehler"
3. Prozent → 2.5 → ein Followup-Save mit empty patch (`Request Body: []`, `content-length:2`)
4. UI-State haelt skonto_percent=null obwohl DB {2.0, 7} hat
5. Page-Reload → korrekter State

**Vermutung:** setState-Race im handleProposalChange/debouncedPersist-Pfad in proposal-editor.tsx. Der Server-Action-Empty-Patch ist auffaellig — eventuell stale Closure des debounce-Callbacks.

**Funktional intakt:** Validation greift (Inline-Error sichtbar), DB-State immer korrekt, kein Datenverlust.

**Fix-Optionen:**
- (A) Bei Save-Error in proposal-editor.tsx den lokalen State explicit revert auf den Pre-Patch-Wert (optimistic-update-Reset)
- (B) Investigation: warum sendet der debounce mit empty patch?

Aufwand: ~30min Investigation + Fix. Backlog: BL-419, KNOWN_ISSUES: ISSUE-049.
