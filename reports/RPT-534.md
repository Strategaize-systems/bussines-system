## Discovery

User-Mail-Postfach-Check 2026-05-23 14:20 UTC nach MIG-040-Fix (RPT-533). Test-Mail aus Composing-Studio kam erfolgreich an:
- Subject: "Danke fuer den Termin — und der naechste Schritt"
- From: bd@strategaizetransition.com
- To: immo@bellaerts.de
- Branding-Footer mit "Richard Bellaerts, Strategaize transition BV, www.strategaizetransition.com, Strategaize Transition GmbH | Am Markt 19 | Swalmen | NL"
- **DSE-Footer fehlt komplett.** Erwarteter Block "Datenschutzerklaerung: https://business.strategaizetransition.com/p/strategaize-transition-bv/datenschutz" ist nicht im HTML.

User-Frage: "Müsste hier nicht Datenschutz oder so erscheinen?"

## Root Cause

**Pre-Render-Bypass:** Compose-Pfad rendert den HTML serverseitig in der Server-Action und reicht ihn als `params.html` an sendEmailWithTracking. Der `if (params.html)`-Shortcut in send.ts umgeht den tenantSlug-Lookup-else-branch.

### Code-Walkthrough

**`cockpit/src/app/(app)/emails/compose/send-action.ts` Zeile 100 (vor Fix):**
```typescript
const html = renderBrandedHtml(body, branding, vars);   // ❌ kein tenantSlug
// ...
const result = await sendEmailWithTracking({
  // ...
  html,                                                  // ❌ pre-rendered → shortcut
  ownerUserId: profile.user_id,                          // ✅ gesetzt, aber wirkungslos
});
```

**`cockpit/src/lib/email/send.ts` Zeile 103-119:**
```typescript
let htmlContent: string;
if (params.html) {
  htmlContent = params.html;                             // ⚠ TRIGGERED fuer Compose-Pfad
} else {
  const branding = await getBrandingForSend();
  const tenantSlug = params.ownerUserId
    ? (await getTenantSlugByOwnerUserId(params.ownerUserId)) ?? undefined
    : undefined;
  htmlContent = renderBrandedHtml(                       // ❌ NEVER REACHED via Compose
    params.body,
    branding,
    params.vars ?? {},
    tenantSlug,
  );
}
```

**Folge:** SLC-846 hat send.ts korrekt gepatched, aber compose/send-action.ts nicht. Der renderBrandedHtml-Aufruf in compose hat keinen `tenantSlug` → Footer-Block wird `""` (siehe render.ts Zeile 73 `if (!tenantSlug) return ""`).

### Warum hat Vitest das nicht catchen koennen?

`render.test.ts` testet `renderBrandedHtml` direkt mit `tenantSlug` als 4. Param gesetzt — 11/11 PASS. Aber: kein Test traversiert den Aufrufer-Site `sendComposedEmail`. Caller-Site-Patch-Gap → unsichtbar fuer Unit-Tests. Integration-Test gegen echte Compose-Form-Send-Action ware noetig gewesen.

Plus: stale Code-Kommentar in send.ts Zeile 102 sagt: `"params.html ueberschreibt den Renderer (heute nirgendwo gesetzt, Backwards Compat)"`. Falsch — sendComposedEmail SETZT params.html in jedem Compose-Send.

## Fix

`cockpit/src/app/(app)/emails/compose/send-action.ts`:

```typescript
import { getTenantSlugByOwnerUserId } from "@/lib/team/lookup-slug";  // NEU

// In sendComposedEmail nach getBrandingForSend + vars-Auflosung:
const tenantSlug = profile.user_id
  ? (await getTenantSlugByOwnerUserId(profile.user_id)) ?? undefined
  : undefined;
const html = renderBrandedHtml(body, branding, vars, tenantSlug);  // 4. Param NEU
```

Code unveraendert in send.ts und render.ts.
