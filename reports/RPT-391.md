## Problems Found
- **P1 — RLS-Helper-Function-Performance unbekannt:** Postgres cached STABLE-Functions pro Statement, aber Production-Verhalten muss gemessen werden. Open Technical Question 3 fuer /slice-planning (PgBench-Smoke vor SLC-701-Release).
- **P2 — owner_user_id NOT NULL Entscheidung:** MIG-035 setzt KEIN NOT NULL, weil System-Records (Cron-Inserts) ohne User-Context NULL bleiben muessen. Konsequenz: Code-Side Checks fuer NULL-Owner muessen in Listing-Queries beruecksichtigt werden (Member SELECT geht via RLS, aber Listing-UI muss "Unzugewiesen"-Label anzeigen koennen).
- **P3 — Bestehende Cron-Jobs sind nicht owner-aware:** Followup-Cron, Briefing-Cron, Auto-Winloss-Cron, Signal-Extract-Cron laufen aktuell mit Service-Role. Erzeugte Records bekommen NULL-Owner. SLC-701 muss Cron-Code anpassen, dass erzeugte Records den Owner des Source-Records erben (z.B. Briefing-Activity bekommt Meeting-Owner). Open Technical Question 7 fuer /slice-planning.
- **P4 — Workflow-Engine V6.2 Owner-Pass nicht implementiert:** Rules feuern aktuell ohne Owner-Context-Pass. Wenn Workflow eine Activity erzeugt, bekommt sie aktuell NULL-Owner (Service-Role-Insert). DEC-185 sagt team-shared, aber konkrete Code-Aenderung in `lib/automation/dispatcher.ts` muss in SLC-704 erfolgen. Open Technical Question 5.
- **P5 — Migration-Reihenfolge auf Hetzner ist sensibel:** Phase A→B→C muss in einer Coolify-Sitzung laufen. Falls C scheitert: zwischen-Stand mit Daten gebackfilled aber ohne RLS-Switch. Code-Side noch im V6.6-Verhalten = funktionsfaehig, aber Cockpit-Daten zeigen owner_user_id ohne dass es genutzt wird. Re-Apply MIG-035 als Recovery. Backout-Test pro Phase in SLC-701 Pflicht. Open Technical Question 2.
- **P6 — ~80 bestehende Server Actions muessen owner-aware werden:** SLC-704 ist eine grosse Anstrengung. Konkretes Audit-Spreadsheet aller Server Actions muss erstellt werden (Insert/Update/Delete, ob Owner-Context schon gesetzt wird, ob ReadOnly-Guard noetig). Empfehlung: SLC-704 startet mit Audit-Liste als ersten MT.
- **P7 — RAG / Knowledge-Chunks haben keinen Owner-Bezug:** DEC-185 sagt team-shared, was den Status-Quo beibehaelt. Falls V7.5 Owner-Filter eingefuehrt wird, muessen alle Chunks re-embedded werden? Antwort in OTQ 6: owner-spalte mit NULL-Default in MIG-033 nicht noetig (Owner-Lookup via source_id), Re-Embedding spaeter nicht noetig. Im V7-Architecture-Doc dokumentiert.
