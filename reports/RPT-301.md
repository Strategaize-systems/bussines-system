### SLC-572 → SkontoMutex-Effect Regression

SkontoMutex-Effect (proposal-editor.tsx:181-190) ruft bei Mutex-Transition `false→true` automatisch `patchAndSave({skonto_percent: null, skonto_days: null})` (Auto-Clear bei 100%-Vorkasse).

- **Auf Success:** `nextSkontoRefAfterSave(prev, {null, null})` → ref aktualisiert auf `{null, null}` ✅. Korrekt.
- **Auf Error (edge case, z.B. assertProposalEditable rejected):** `revertPatchIfSkontoFailed({null, null}, lastGood={2.0, 7})` → returns `{2.0, 7}`. Dann `onProposalChange({2.0, 7})` → State zurueck auf {2.0, 7}. **Aber:** SkontoSection.disabled ist via `skontoMutex` true, also Toggle bleibt visually disabled — Werte 2.0/7 werden zwar im State gehalten, aber UI zeigt sie wegen `skontoMutex && !readonly` Banner als "Skonto deaktiviert". Edge-Case, unwahrscheinlich, kein Blocker.

**Verdict:** Mutex-Auto-Clear-Pfad bleibt funktional, mit begruendbarem Edge-Case-Verhalten.

#### **L1 — Reverse-Charge-Toggle UI-State-Drift (analog ISSUE-049 fuer Skonto)** — BL-422 angelegt
