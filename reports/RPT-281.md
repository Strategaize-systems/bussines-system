## Problems found

- **Audit-Taxonomie-Drift:** `AuditAction`-Type in `lib/audit.ts` enthaelt nur 5 Actions (stage_change|status_change|create|update|delete). Cron nutzt 3 neue String-Actions (`ai_briefing_generated`/`ai_briefing_retry_armed`/`ai_briefing_failed_permanently`) via direkten Insert (kein logAudit-Call), umgeht damit die Type-Pruefung. Konsistent mit V5.1+ Pattern (recording-retention nutzt auch direkten Insert), aber Audit-Type-Erweiterung waere sauberer. **IMP-Kandidat fuer Skill-Improvements.**
- **Sentinel-Lookup via LIKE-on-JSON:** Failure-Count via `LIKE '%"meetingId":"…"%'` auf `activities.description`-JSON. Funktioniert (Index-Scan via deal_id), aber String-fragil (Format-Drift im JSON wuerde brechen). Alternativen waeren JSONB-Cast oder ein dedizierter `briefing_error`-Counter-Spalte auf `meetings`. Bei aktueller Anzahl Errors kein Problem; bei Skalierung pruefen. **IMP-Kandidat.**
- **Deal-Notes/Probability-Drift:** Slice spec referenziert `deal.notes` + `deal.probability`. DB-Schema hat keine `notes`-Spalte auf deals; `probability` ist auf `pipeline_stages` (Stage-Default) statt deal-spezifisch. Loader liefert `notes=undefined` und nutzt Stage-Probability — ist ein milder Loss-of-Info ggü. dem Slice-Vorhaben, aber Schema-konform. Kein Action-Item, nur Doku-Hinweis.
- **Single-User-Annahme im Cron:** Cron nimmt erste user_settings-Row (`limit(1).maybeSingle()`). Bei Multi-User-Erweiterung (V7) muss der Cron pro User iterieren. PRD-V5.6 + Roadmap haben das bereits als V7-Scope markiert.
- **Coolify-Cron-Anlage-Schritt:** Ohne diesen User-Aktion-Schritt nach Deploy bleibt das Briefing **stumm** (Cron-Endpoint live, aber kein Trigger). REL-022-Notes haben die Anleitung. Bei /go-live muss der User das **vor erstem Live-Test** machen.
