## Findings

### Blocker (gefixt)

#### 1. Server-Mixed-Content via createSignedUrl auf interner Kong-URL
- **Found in:** Block 1 Re-Smoke nach erstem PDF-Generate
- **Root cause:** `admin.storage.from(...).createSignedUrl(...)` returnt URL mit Kong-Container-Hostname `http://supabase-kong:8000`. Coolify-Reverse-Proxy hat KEIN Routing zu Kong; externe URL `https://.../supabase` liefert 404. Browser blockiert die HTTP-iframe-URL ausserdem mit Mixed-Content-Error.
- **Impact:** PDF-Generierung war serverseitig vollstaendig erfolgreich (Storage + DB + audit_log + Renderer-Performance), aber iframe-Anzeige + Download im Modal komplett broken. Nicht-trivial reproduzierbar nur durch echten externen Browser-Zugriff.
- **Fix:** Hotfix Commit `91020b2` (push 2026-04-30 10:21):
  - Neue Route `cockpit/src/app/api/proposals/[id]/pdf/route.ts` (Pattern analog SLC-531 `/api/branding/logo`)
  - `generateProposalPdf` returnt relative URL `/api/proposals/{id}/pdf?v={version}-{timestamp}` statt Signed-URL
  - `createSignedUrl` + `PDF_SIGNED_URL_TTL_SECONDS` entfernt
  - Cache-Buster verhindert iframe-Re-Use bei erneuter Generierung
  - 76/76 Tests gruen, Build sauber
- **Verification post-fix:** Re-Smoke lieferte HTTP 200, application/pdf, 80131 Bytes, Magic %PDF, Content-Disposition korrekt.
- **Konsistenz mit Memory-Feedback:** `feedback_no_browser_supabase.md` warnt explizit vor Browser-side Supabase-URLs — der Bug wiederholt das Pattern. Adapter-Pattern bleibt der richtige Weg.
